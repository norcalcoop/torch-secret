  <action>
**Root cause:** `createAnonHourlyLimiter` uses `standardHeaders: 'draft-6'`, which emits
`RateLimit-Reset` as a delta in seconds (e.g. 1570 = seconds until window resets). The current
`showRateLimitUpsell()` code treats it as a Unix timestamp:

```typescript
// WRONG — treats 1570 as epoch seconds → year 1970 → minutesUntilReset always negative
const resetMs = resetTimestamp * 1000;
const minutesUntilReset = Math.ceil((resetMs - Date.now()) / 60_000);
```

**Fix in `client/src/pages/create.ts` — `showRateLimitUpsell()` function (around line 186):**

Replace the three lines:
```typescript
const resetMs = resetTimestamp * 1000;
const minutesUntilReset = Math.ceil((resetMs - Date.now()) / 60_000);
const resetText =
```

With:
```typescript
const minutesUntilReset = Math.ceil(resetTimestamp / 60);
const resetText =
```

The value is already seconds-remaining, so dividing by 60 and ceiling gives minutes remaining
directly. No epoch arithmetic needed.

The outer condition `if (resetTimestamp && resetTimestamp > 0)` already guards against
undefined/zero, so `minutesUntilReset` will always be a positive integer inside that block
when the header is present with a valid value. The `minutesUntilReset > 0` check in the
ternary remains correct as a belt-and-suspenders guard for edge cases (e.g. resetTimestamp
is 1–59 seconds, which rounds up to 1 minute — correct behavior).

**Fix in `client/src/api/client.ts` — JSDoc on `ApiError.rateLimitReset` field (line 27):**

Change:
```typescript
readonly rateLimitReset?: number; // Unix timestamp (seconds) from RateLimit-Reset header
```

To:
```typescript
readonly rateLimitReset?: number; // Delta in seconds (time remaining) from RateLimit-Reset draft-6 header
```

Also update the JSDoc block comment on `createSecret()` that says "Unix timestamp seconds"
to say "delta in seconds (time remaining)" to match the corrected field description.
