  <action>
Create the client/public/ directory if it does not exist, then create these 4 files:

**1. client/public/favicon.svg** — Adaptive filled shield icon:
- ViewBox: 0 0 32 32
- Design a simple filled shield silhouette optimized for 16x16-32x32 rendering. Do NOT use the Lucide stroke-based path directly — it loses definition at small sizes. Instead, create a clean filled shield shape: a symmetrical shield with a slight point at the bottom, filled solidly (no stroke, no outlines, no interior detail).
- Embed a `<style>` block with `.shield { fill: #4d8bf5; }` for light mode (the accent blue from the design system, --ds-color-accent: oklch(0.71 0.143 255), approximately #4d8bf5 in hex).
- Add `@media (prefers-color-scheme: dark) { .shield { fill: #7fb3ff; } }` for a lighter blue variant visible on dark tab backgrounds.
- Apply class="shield" to the path element. No stroke attributes.
- The SVG should be minimal — no unnecessary metadata, comments, or XML processing instructions beyond the xmlns.

**2. client/public/robots.txt** — Crawl policy (per user decision):
```
User-agent: *
Allow: /
Disallow: /api/
Disallow: /secret/

Sitemap: https://secureshare.example.com/sitemap.xml
```
Exactly this content. The /secret/ disallow is a locked user decision. The placeholder domain is a locked decision.

**3. client/public/sitemap.xml** — Homepage-only sitemap:
```xml
<?xml version="1.0" encoding="UTF-8"?>
<urlset xmlns="http://www.sitemaps.org/schemas/sitemap/0.9">
  <url>
    <loc>https://secureshare.example.com/</loc>
  </url>
</urlset>
```
No `changefreq` or `priority` (Google ignores them). Use the placeholder domain per user decision.

**4. client/public/site.webmanifest** — Web app manifest:
```json
{
  "name": "SecureShare",
  "short_name": "SecureShare",
  "description": "Share secrets securely with zero-knowledge, one-time encrypted links",
  "start_url": "/",
  "display": "browser",
  "background_color": "#1a1625",
  "theme_color": "#1a1625",
  "icons": [
    {
      "src": "/favicon.svg",
      "type": "image/svg+xml",
      "sizes": "any"
    },
    {
      "src": "/apple-touch-icon.png",
      "type": "image/png",
      "sizes": "180x180"
    }
  ]
}
```
- display: "browser" (NOT "standalone") — SecureShare is a URL-sharing tool, not a PWA. Users need the browser URL bar.
- background_color and theme_color: #1a1625 — the hex equivalent of --ds-color-bg: oklch(0.23 0.038 283), the deep navy-black from the design system.
- Only reference icons that actually exist (favicon.svg and apple-touch-icon.png). Do NOT include 192x192 or 512x512 entries that don't have corresponding files.

<task type="auto">
  <name>Task 2: Generate favicon.ico and apple-touch-icon.png from SVG</name>
  <files>scripts/generate-favicons.mjs, client/public/favicon.ico, client/public/apple-touch-icon.png</files>
  <action>
Create a one-time Node.js script at `scripts/generate-favicons.mjs` that converts the SVG favicon into favicon.ico (32x32) and apple-touch-icon.png (180x180).

**Approach decision:** Since no image processing library (sharp, canvas) is installed and adding one as a devDependency for a one-time operation is over-engineering (per research recommendations), use one of these approaches in order of preference:

**Option A (preferred): Programmatic generation without external dependencies**
Write a Node.js script that:
1. Reads client/public/favicon.svg
2. For favicon.ico: Create a 32x32 ICO file programmatically. The ICO format is a simple binary container: a 6-byte header (reserved=0, type=1, count=1), a 16-byte directory entry (width=32, height=32, colors=0, reserved=0, planes=1, bitcount=32, size, offset), followed by a BMP or PNG payload. The simplest valid ICO wraps a PNG image. Since we don't have a PNG renderer, generate the ICO by:
   - Using a canvas-free approach: create a minimal valid ICO that wraps a small embedded PNG
   - OR: if programmatic generation proves too complex without dependencies, fall back to Option B

3. For apple-touch-icon.png: Similarly needs SVG-to-PNG rasterization at 180x180.

**Option B (fallback): One-time devDependency script**
If Option A is not feasible:
1. Temporarily install sharp as a devDependency: `npm install --save-dev sharp`
2. Write the script using sharp to convert SVG to ICO (32x32) and PNG (180x180)
3. Run the script to generate the files
4. Uninstall sharp: `npm uninstall sharp`
5. The generated files remain committed; the dependency does not

**Option C (last resort): Manual base64 embedding**
If neither programmatic approach works cleanly, generate simple solid-color shield favicon.ico and apple-touch-icon.png as pre-computed base64 binary data embedded in the script, decode and write to files. This is acceptable because the favicon rarely changes.

Regardless of approach:
- The script MUST be runnable with `node scripts/generate-favicons.mjs`
- Output files: `client/public/favicon.ico` (32x32) and `client/public/apple-touch-icon.png` (180x180)
- After running, verify both files exist and have non-zero size
- The generated files should be committed to git (they are static assets, not build artifacts)
- Add a comment at the top of the script explaining it is a one-time generation tool, not part of the build pipeline
  </action>

  </verify>
