<task type="auto">
  <name>Task 1: Add favicon, manifest, and apple-touch-icon link tags to HTML head</name>
  <files>client/index.html</files>
  <action>
Read client/index.html and add the following link tags inside `<head>`, after the existing `<title>` tag and before the stylesheet link:

```html
<link rel="icon" href="/favicon.ico" sizes="32x32">
<link rel="icon" href="/favicon.svg" type="image/svg+xml">
<link rel="apple-touch-icon" href="/apple-touch-icon.png">
<link rel="manifest" href="/site.webmanifest">
```

Critical details:
- The ICO link MUST include `sizes="32x32"` — this is needed to work around a Chrome bug where Chrome prefers ICO over SVG without this hint (see research Pitfall 1).
- Order matters: ICO first (with sizes constraint), then SVG (preferred by modern browsers), then apple-touch-icon, then manifest.
- Do NOT use `rel="shortcut icon"` — the `shortcut` keyword is non-standard and unnecessary.
- Do NOT add any other favicon link tags (no 16x16, 48x48, 64x64 PNG variants — the three-file approach covers all browsers).

After the changes, the `<head>` section should look like:
```html
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <meta name="referrer" content="no-referrer" />
  <title>SecureShare</title>
  <link rel="icon" href="/favicon.ico" sizes="32x32">
  <link rel="icon" href="/favicon.svg" type="image/svg+xml">
  <link rel="apple-touch-icon" href="/apple-touch-icon.png">
  <link rel="manifest" href="/site.webmanifest">
  <link rel="stylesheet" href="/src/styles.css" />
</head>
```
  </action>

<task type="auto">
  <name>Task 2: Add noscript fallback content inside #app container</name>
  <files>client/index.html</files>
  <action>
Add a `<noscript>` block inside the `<div id="app">` element. The noscript content goes INSIDE the #app div because:
1. The app's JavaScript will replace #app's innerHTML on load, so the noscript content naturally disappears when JS runs
2. Placing noscript in the body but outside #app would leave it permanently visible (even with JS)
3. Noscript in `<head>` cannot contain visible content (HTML spec limitation)

Add this content inside `<div id="app" class="max-w-2xl mx-auto px-4 py-8">`:

```html
<noscript>
  <div style="max-width: 36rem; margin: 2rem auto; padding: 1.5rem; font-family: system-ui, -apple-system, sans-serif; color: #e8e4f0; line-height: 1.6;">
    <h1 style="font-size: 1.5rem; font-weight: 700; margin-bottom: 1rem; color: #f0ecf8;">SecureShare</h1>
    <p style="margin-bottom: 0.75rem;">
      Share secrets securely with zero-knowledge, one-time encrypted links.
      Your secret is encrypted entirely in your browser before it ever reaches our server.
    </p>
    <p style="margin-bottom: 0.75rem;">
      <strong>How it works:</strong> Paste your secret, get an encrypted link, share it.
      The recipient views it once, then it's permanently destroyed. No accounts required.
    </p>
    <p style="margin-bottom: 0.75rem; padding: 0.75rem; border-left: 3px solid #4d8bf5; background: rgba(77, 139, 245, 0.1);">
      <strong>Security:</strong> AES-256-GCM encryption. The encryption key lives only in the URL fragment
      and never reaches the server. Even a complete database breach reveals nothing.
    </p>
    <p style="color: #a89ec4; font-size: 0.875rem; margin-top: 1rem;">
      JavaScript is required to use SecureShare. Encryption and decryption
      happen entirely in your browser using the Web Crypto API.
    </p>
  </div>
</noscript>
```

IMPORTANT details:
- Use ONLY inline styles, NOT Tailwind classes. Tailwind CSS is loaded via JavaScript (Vite). When JS is disabled, Tailwind classes have no effect and the content would be unstyled.
- Color values approximate the dark theme: #e8e4f0 for body text (~text-primary on dark bg), #f0ecf8 for heading, #a89ec4 for muted text, #4d8bf5 for accent.
- The existing body has `class="min-h-screen bg-bg text-text-primary"` which relies on Tailwind — but the page `background-color` and `color` won't apply without JS either. The body will render with browser defaults (white bg, black text) when JS is off. The noscript div uses its own inline dark-ish colors, but on a white background they'll still be readable because the colors are light/pastel. Accept this as a graceful degradation — the content is readable either way.
- Include: app description, how-it-works summary, security highlights, and a clear JS-required notice.
- The final index.html `<div id="app">` should look like:

```html
<div id="app" class="max-w-2xl mx-auto px-4 py-8">
  <noscript>
    <!-- noscript content here -->
  </noscript>
</div>
```
