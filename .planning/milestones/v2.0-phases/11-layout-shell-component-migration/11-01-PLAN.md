<task type="auto">
  <name>Task 2: Wire layout shell into app.ts and add routechange event to router.ts</name>
  <files>client/src/app.ts, client/src/router.ts</files>
  <action>
**In `client/src/app.ts`:**
- Add import: `import { createLayoutShell } from './components/layout.js';`
- In the DOMContentLoaded handler, call `createLayoutShell()` BEFORE `initRouter()`:
```typescript
document.addEventListener('DOMContentLoaded', () => {
  createLayoutShell();
  initRouter();
});
```

**In `client/src/router.ts`:**
- At the END of the `handleRoute()` function (after the if/else routing block), dispatch a custom event:
```typescript
window.dispatchEvent(new CustomEvent('routechange', {
  detail: { path: window.location.pathname }
}));
```
- This fires on EVERY navigation: initial load (via `initRouter` -> `handleRoute`), popstate (back/forward), and programmatic navigation (via `navigate` -> `handleRoute`). This ensures the header's "Create" link visibility updates on all navigation types.

**Why this order matters:** `createLayoutShell()` must run before `initRouter()` because:
1. Layout shell sets up the header's `routechange` event listener
2. `initRouter()` calls `handleRoute()` which dispatches `routechange`
3. If router fires first, the header listener doesn't exist yet and misses the initial route
