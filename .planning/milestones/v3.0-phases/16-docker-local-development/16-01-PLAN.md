<task type="auto">
  <name>Task 1: Create health check endpoint and mount before rate-limited routes</name>
  <files>server/src/routes/health.ts, server/src/app.ts</files>
  <action>
Create `server/src/routes/health.ts` as an Express Router with a single GET `/` handler:

1. Import `Router` from express and `pool` from `../db/connection.js`
2. Create and export a `healthRouter` using `Router()`
3. The GET `/` handler should:
   - Build a health object with `status: 'ok'`, `uptime: process.uptime()`, `timestamp: new Date().toISOString()`
   - Try `await pool.query('SELECT 1')` -- on success set `database: 'connected'`, on catch set `database: 'disconnected'` and `status: 'degraded'`
   - Optionally check Redis if available (import `env` from config, check if `env.REDIS_URL` is set -- if so, note Redis status as 'configured' without actively pinging since the Redis client is scoped to buildApp; if not set, `redis: 'not configured'`)
   - Return 200 if status is 'ok', 503 if 'degraded'
   - Response shape: `{ status, database, redis, uptime, timestamp }`
4. Use proper TypeScript types -- the handler is `async`

Mount in `server/src/app.ts`:
- Import `healthRouter` from `./routes/health.js`
- Mount `app.use('/api/health', healthRouter)` AFTER the JSON parser (`express.json()`) but BEFORE the secrets router (`app.use('/api/secrets', ...)`)
- This placement means the health endpoint gets trust proxy, HTTPS redirect, CSP nonce, helmet, and logger (all harmless for health checks) but is NOT rate-limited (rate limiting is inside the secrets router factory)
- Add a comment explaining why it's placed before secrets routes: "Health check -- before rate-limited routes"

Do NOT create a separate middleware or complex abstraction. This is a simple route with a DB ping.
  </action>

  </verify>

<task type="auto">
  <name>Task 2: Move tsx to production dependencies and create .dockerignore</name>
  <files>package.json, .dockerignore</files>
  <action>
**package.json changes:**
1. Move `tsx` from `devDependencies` to `dependencies` in package.json
2. The version constraint stays the same (`"^4.21.0"`)
3. Do NOT run `npm install` -- just move the entry between the two objects in the JSON. The lockfile already has tsx resolved and it will be installed correctly by `npm ci` in either section.

**Why:** The production Docker image runs `npm ci --omit=dev` and needs tsx to execute `node --import tsx server/src/server.ts`. tsx is ~2MB and eliminates the need for a separate TypeScript compilation step for the server.

**.dockerignore:**
Create `.dockerignore` at the project root with these entries:

```
node_modules
.git
.gitignore
.planning
.playwright-mcp
.husky
.env
.env.*
!.env.example
dist
client/dist
*.png
*.md
!package.json
.DS_Store
.claude
.claude.local.md
CLAUDE.md
coverage
```

**Why each exclusion:**
- `node_modules`: Platform-specific native binaries (macOS vs Linux); always install inside container
- `.git`: Large history unnecessary for build
- `.planning`: GSD workflow docs, not needed for app
- `.husky`, `.playwright-mcp`: Dev tooling
- `.env`, `.env.*`: Secrets (but keep .env.example as reference)
- `dist`, `client/dist`: Build outputs (rebuilt inside container)
- `*.png`, `*.md`, `CLAUDE.md`: Documentation and images
- `.claude*`: Claude Code config
- `coverage`: Test coverage reports
  </action>
