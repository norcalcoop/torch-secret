**Create Dockerfile at project root:**

```dockerfile
# ============================================================
# Stage 1: Install ALL dependencies (dev + prod)
# Needed for: Vite frontend build, native modules (argon2)
# ============================================================
FROM node:24-slim AS deps
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci

# ============================================================
# Stage 2: Build frontend assets with Vite
# ============================================================
FROM deps AS build
WORKDIR /app
COPY . .
RUN npm run build:client

# ============================================================
# Stage 3: Production image (minimal)
# - Only production dependencies (no vite, eslint, vitest)
# - Non-root user (node, UID 1000)
# - Includes tsx for running TypeScript server directly
# ============================================================
FROM node:24-slim AS production
ENV NODE_ENV=production
WORKDIR /app

# Install production dependencies only
COPY package.json package-lock.json ./
RUN npm ci --omit=dev && npm cache clean --force

# Copy server source (TypeScript, executed via tsx at runtime)
COPY server/ ./server/
COPY shared/ ./shared/
COPY tsconfig.json ./
COPY server/tsconfig.json ./server/tsconfig.json

# Copy Drizzle config and migrations for db:migrate
COPY drizzle.config.ts ./
COPY drizzle/ ./drizzle/

# Copy built frontend from build stage
COPY --from=build /app/client/dist ./client/dist

# Docker HEALTHCHECK (node:24-slim has no curl/wget)
HEALTHCHECK --interval=30s --timeout=10s --start-period=15s --retries=3 \
  CMD node -e "fetch('http://localhost:3000/api/health').then(r=>{if(!r.ok)throw new Error()}).catch(()=>process.exit(1))"

# Switch to non-root user (built into node images, UID 1000)
USER node

EXPOSE 3000
CMD ["node", "--import", "tsx", "server/src/server.ts"]
```

**Key decisions:**
- `node:24-slim` base (Debian Bookworm) -- compatible glibc for argon2 prebuilt binaries, avoids Alpine musl issues
- Stage 1 `npm ci` installs ALL deps for Vite build; Stage 3 `npm ci --omit=dev` installs production only
- tsx is in production dependencies (moved in Plan 01) so it's available in Stage 3
- `server/tsconfig.json` is copied separately because it extends the root tsconfig and tsx needs it for NodeNext resolution
- HEALTHCHECK uses `node -e fetch()` since slim image has no curl
- HEALTHCHECK port is hardcoded to 3000 (Docker's internal port, not Render's 10000)
- USER node runs after all COPY commands (node user can't write to /app since WORKDIR was created as root)
- drizzle.config.ts is copied for `npm run db:migrate` (drizzle-kit reads it)
- No ENTRYPOINT -- CMD can be overridden by docker-compose or Render's dockerCommand

**Important:** Do NOT copy the root `drizzle.config.ts` via a wildcard. Copy it explicitly. Do NOT include `.env` or `.env.example` -- env vars come from Docker Compose or Render environment.

<task type="auto">
  <name>Task 2: Create docker-compose.yml for one-command local development</name>
  <files>docker-compose.yml</files>
  <action>
Create `docker-compose.yml` at the project root:

```yaml
services:
  db:
    image: postgres:17-alpine
    environment:
      POSTGRES_USER: secureshare
      POSTGRES_PASSWORD: secureshare
      POSTGRES_DB: secureshare
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U secureshare"]
      interval: 5s
      timeout: 5s
      retries: 5
      start_period: 10s

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 5s
      timeout: 5s
      retries: 5

  app:
    build: .
    init: true
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgresql://secureshare:secureshare@db:5432/secureshare
      REDIS_URL: redis://redis:6379
      NODE_ENV: production
      PORT: "3000"
      LOG_LEVEL: info
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy

volumes:
  pgdata:
```

**Key decisions:**
- `init: true` on the app service enables tini as PID 1 for proper signal handling (Node.js doesn't handle SIGTERM well as PID 1)
- `depends_on` with `condition: service_healthy` ensures PostgreSQL and Redis are accepting connections before the app starts
- PostgreSQL healthcheck uses `pg_isready` (built into postgres image)
- Redis healthcheck uses `redis-cli ping` (built into redis image)
- `pgdata` named volume persists database data across `docker compose down`/`up` cycles
- Port 5432 and 6379 are exposed to host so developer can also connect tools (psql, redis-cli) directly
- `NODE_ENV: production` because the Docker image serves the built frontend (no Vite dev server)
- Database credentials are simple for local dev (secureshare/secureshare) -- production uses Render's managed credentials
- No `restart: unless-stopped` -- this is for local development, not production orchestration

**Database migration handling:**
The app container needs to run migrations before serving traffic. The approach depends on whether `drizzle-kit migrate` can run programmatically on startup or needs to be a separate step.

Option A (recommended): Override CMD in docker-compose to run migrations then start:
```yaml
    command: ["sh", "-c", "npx drizzle-kit migrate && node --import tsx server/src/server.ts"]
```

Option B: Use `npm run db:migrate` which calls the migrate.ts script.

Use Option A -- add a `command` to the app service that runs migrations before the server starts. This ensures the secrets table exists on first `docker compose up`. The `sh -c` wrapper is needed to chain commands.

Update the app service to include:
```yaml
    command: ["sh", "-c", "npx drizzle-kit migrate && node --import tsx server/src/server.ts"]
```

Note: This overrides the Dockerfile's CMD. In production (Render), the dockerCommand in render.yaml will do the same thing. The Dockerfile CMD is the fallback for `docker run` without compose.
