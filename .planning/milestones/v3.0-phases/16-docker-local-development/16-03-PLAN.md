Create `render.yaml` at the project root with the following structure:

```yaml
# Render.com Blueprint — one-click deployment for SecureShare
# Deploy: https://render.com/deploy → import this repo
#
# IMPORTANT: Free tier PostgreSQL expires after 30 days (14-day grace period).
# For persistent production use, upgrade to a paid database plan.
# Free tier Redis (keyvalue) data is ephemeral and lost on restart.
# Free web services spin down after 15 minutes of inactivity.

services:
  # Web service — Docker-based Node.js application
  - type: web
    name: secureshare
    runtime: docker
    plan: free
    region: oregon
    healthCheckPath: /api/health
    dockerCommand: "sh -c 'npx drizzle-kit migrate && node --import tsx server/src/server.ts'"
    envVars:
      - key: DATABASE_URL
        fromDatabase:
          name: secureshare-db
          property: connectionString
      - key: REDIS_URL
        fromService:
          type: keyvalue
          name: secureshare-redis
          property: connectionString
      - key: NODE_ENV
        value: production
      - key: LOG_LEVEL
        value: info

  # Redis key-value store for distributed rate limiting
  - type: keyvalue
    name: secureshare-redis
    plan: free
    region: oregon
    ipAllowList: []  # Only accessible from Render internal network
    maxmemoryPolicy: allkeys-lru

# PostgreSQL database
databases:
  - name: secureshare-db
    plan: free
    region: oregon
    postgresMajorVersion: "17"
```

**Key decisions and reasons:**
- `runtime: docker` tells Render to build using the project's Dockerfile (not a native Node.js buildpack)
- `dockerCommand` overrides the Dockerfile CMD to run migrations before the server starts. This is necessary because Render free tier does not support `preDeployCommand`. The command uses `sh -c` to chain migrate + start.
- `healthCheckPath: /api/health` -- Render pings this path to determine if the service is healthy after deploy
- Do NOT set `PORT` in envVars -- Render automatically provides PORT=10000, and the app's Zod env config (`PORT: z.coerce.number().default(3000)`) will use the Render-provided value
- `fromDatabase` wires `DATABASE_URL` automatically from the managed PostgreSQL instance (includes host, user, password, database name)
- `fromService` with `type: keyvalue` wires `REDIS_URL` from the managed Redis instance
- `type: keyvalue` (NOT `type: redis` which is deprecated)
- `ipAllowList: []` restricts Redis to Render's internal network only
- `maxmemoryPolicy: allkeys-lru` evicts least-recently-used keys when memory is full (appropriate for rate-limiting cache)
- `region: oregon` -- all three services in the same region to minimize latency. Oregon is Render's default and has the best free tier availability.
- `postgresMajorVersion: "17"` -- matches the project requirement and docker-compose setup
- Comments document the free tier limitations prominently at the top of the file

**Validation notes:**
- The `render.yaml` schema is documented at https://render.com/docs/blueprint-spec
- The `fromDatabase` property name is `connectionString` (not `url`)
- The `fromService` needs `type: keyvalue` and `property: connectionString`
- Do NOT include `docker:` or `dockerfilePath:` -- Render auto-discovers the Dockerfile at the root
