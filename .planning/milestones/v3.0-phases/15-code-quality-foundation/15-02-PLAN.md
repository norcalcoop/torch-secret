The fix is a behavioral correctness fix AND a type fix: convert the `class` value from `string[]` to a space-joined `string` BEFORE passing to `createElement`.

   Find where the `class` property is set on the `attrs` object (it will be an array like `['lucide', 'lucide-icon-name']`). Join it into a single string:
   ```typescript
   // Where classValue is the string[] being assigned to attrs.class:
   attrs.class = classValue.join(' ');
   ```

   Then the `createElement` call needs no type assertion — `attrs` will naturally be `Record<string, string | number>` once no `string[]` values remain. If the attrs type annotation includes `string[]`, narrow it:
   ```typescript
   const finalAttrs: Record<string, string | number> = {
     ...attrs,
     class: (attrs.class as string[]).join(' '),
   };
   return createElement(icon, finalAttrs) as SVGSVGElement;
   ```

   This both satisfies the type checker AND produces the correct SVG `class` attribute format (space-separated string, not an array).

2. Lines 37, 99, 122 — `toHaveNoViolations` is not recognized on the `Assertion` type.

Fix requires TWO parts — (a) runtime matchers import and (b) type augmentation:

**Part (a): Fix the runtime import in accessibility.test.ts**

The current import gets `toHaveNoViolations` from `vitest-axe/matchers`, but vitest-axe may export it as type-only from that path. Change the import to get the runtime matchers object:
```typescript

expect.extend(matchers);
```
This passes the entire matchers object (which is a runtime value) to `expect.extend`, avoiding the "cannot be used as a value" error on the named export.

If `vitest-axe/matchers` still only exports types, use the main entry point:
