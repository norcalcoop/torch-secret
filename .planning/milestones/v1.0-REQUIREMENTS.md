# Requirements: SecureShare v1.0

**Defined:** 2026-02-13
**Core Value:** Users can share sensitive information once, securely, without accounts or complexity

## v1 Requirements

All requirements shipped and validated in v1.0.

### Encryption

- [x] **ENCR-01**: Secret is encrypted client-side using AES-256-GCM before leaving the browser
- [x] **ENCR-02**: Encryption key is generated per-secret using crypto.getRandomValues (256-bit)
- [x] **ENCR-03**: Encryption key is embedded in URL fragment and never sent to the server
- [x] **ENCR-04**: Unique 96-bit IV is generated per encryption operation
- [x] **ENCR-05**: Ciphertext is padded to fixed block sizes to prevent length leakage

### Secret Creation

- [x] **CREA-01**: User can paste secret text into a textarea (max 10,000 characters)
- [x] **CREA-02**: User can select expiration time (1 hour, 24 hours, 7 days, 30 days; default 24 hours)
- [x] **CREA-03**: User can optionally set a password for the secret
- [x] **CREA-04**: User receives a unique shareable URL after creation
- [x] **CREA-05**: User sees confirmation page with copyable link and expiration details

### Secret Retrieval

- [x] **RETR-01**: Recipient can view the secret by clicking the shared link
- [x] **RETR-02**: Secret is decrypted client-side using the key from the URL fragment
- [x] **RETR-03**: Secret is permanently destroyed on the server after first view (atomic delete)
- [x] **RETR-04**: Subsequent visits to the same link show "secret already viewed or does not exist"
- [x] **RETR-05**: Two-step retrieval: initial page load serves app shell only, explicit user action fetches ciphertext
- [x] **RETR-06**: URL fragment is stripped from browser URL bar immediately after reading the key

### Password Protection

- [x] **PASS-01**: Password-protected secrets require password entry before revealing
- [x] **PASS-02**: Password is verified server-side using secure hashing (Argon2id)
- [x] **PASS-03**: 3 failed password attempts auto-destroy the secret
- [x] **PASS-04**: User sees remaining attempt count during password entry
- [x] **PASS-05**: Password verification uses constant-time comparison

### Expiration

- [x] **EXPR-01**: Background job runs every 5 minutes to delete expired secrets
- [x] **EXPR-02**: Expired secrets return clear "This secret has expired" message
- [x] **EXPR-03**: Expiration timestamp is shown on the creation confirmation page

### Copy & Sharing

- [x] **COPY-01**: One-click copy button for the generated share link (sender side)
- [x] **COPY-02**: One-click copy button for the revealed secret text (receiver side)
- [x] **COPY-03**: Visual confirmation shown when content is copied ("Copied!")

### Security

- [x] **SECR-01**: Server never sees plaintext secrets (zero-knowledge architecture)
- [x] **SECR-02**: Strict Content Security Policy with nonce-based scripts (no unsafe-inline)
- [x] **SECR-03**: Rate limiting: max 10 secret creations per IP per hour
- [x] **SECR-04**: HTTPS enforced with HSTS headers
- [x] **SECR-05**: Referrer-Policy set to no-referrer on all responses
- [x] **SECR-06**: Strict same-origin CORS policy
- [x] **SECR-07**: All API error responses are identical for not-found/expired/wrong-password to prevent enumeration
- [x] **SECR-08**: Secrets are overwritten with zeros before deletion (data remanence mitigation)
- [x] **SECR-09**: No secrets, IDs, or PII logged in application logs

### User Interface

- [x] **UXUI-01**: Mobile-responsive layout that works on phones, tablets, and desktop
- [x] **UXUI-02**: Clear error states: already viewed, expired, invalid password, rate limited, invalid link
- [x] **UXUI-03**: "How it works" trust section explaining zero-knowledge architecture in plain language
- [x] **UXUI-04**: Landing page with textarea, optional settings (collapsed), and "Create Secure Link" button
- [x] **UXUI-05**: Accessible design following WCAG 2.1 AA guidelines (keyboard nav, screen reader support, contrast)

## Requirement Outcomes

All 41 v1 requirements were implemented as specified. Notable outcomes:
- **PASS-02**: Implemented with Argon2id (not bcrypt) per OWASP recommendation
- **ENCR-05**: PADME algorithm chosen over power-of-2 padding (max 12% vs 100% overhead)
- **SECR-03**: Upgraded from MemoryStore to Redis-backed rate limiting in Phase 8
