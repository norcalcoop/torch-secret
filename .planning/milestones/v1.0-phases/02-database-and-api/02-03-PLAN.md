**Test cases (map to Phase 2 success criteria):**

**Success Criterion 1: POST /api/secrets stores and returns ID**
```
test('POST /api/secrets -- creates secret and returns 21-char ID')
  - POST with { ciphertext: 'dGVzdCBjaXBoZXJ0ZXh0', expiresIn: '24h' }
  - Expect 201
  - Expect body.id to have length 21
  - Expect body.expiresAt to be a valid ISO date string
```

**Success Criterion 1b: Validation**
```
test('POST /api/secrets -- rejects missing ciphertext with 400')
  - POST with { expiresIn: '24h' } (no ciphertext)
  - Expect 400 with error: 'validation_error'

test('POST /api/secrets -- rejects invalid expiresIn with 400')
  - POST with { ciphertext: 'dGVzdA==', expiresIn: '2h' } (invalid option)
  - Expect 400 with error: 'validation_error'

test('POST /api/secrets -- rejects empty ciphertext with 400')
  - POST with { ciphertext: '', expiresIn: '24h' }
  - Expect 400 with error: 'validation_error'
```

**Success Criterion 2: GET returns ciphertext exactly once, then deletes**
```
test('GET /api/secrets/:id -- returns ciphertext on first retrieval')
  - Create a secret via POST
  - GET /api/secrets/{id}
  - Expect 200
  - Expect body.ciphertext to equal the original ciphertext

test('GET /api/secrets/:id -- returns 404 on second retrieval (atomic delete)')
  - Create a secret via POST
  - First GET -- expect 200
  - Second GET -- expect 404
```

**Success Criterion 3: Identical error responses (anti-enumeration)**
```
test('error response for consumed secret matches nonexistent secret exactly')
  - Create a secret via POST
  - First GET (consume it) -- 200
  - Second GET (consumed) -- capture response body
  - GET a completely made-up ID (21 chars, like 'xxxxxxxxxxxxxxxxxxx01') -- capture response body
  - Assert: consumed response body deep-equals nonexistent response body
  - Both should be { error: 'not_found', message: 'This secret does not exist, has already been viewed, or has expired.' }
```

**Success Criterion 4: Ciphertext zeroed before deletion (verified indirectly)**
```
test('secret is fully destroyed after retrieval -- no trace remains')
  - Create a secret via POST
  - Retrieve via GET (200)
  - Verify row no longer exists: query secrets table directly via db.select() where id matches -- expect empty result
  (Note: We cannot directly observe the zero-overwrite step since the row is deleted. But the transaction ensures zero happens before delete. The test proves the row is gone, and the service code is reviewed for the zero step.)
```

**Success Criterion 5: No secret IDs in logs (structural verification)**
```
test('logger redacts secret IDs from URL paths')
  - This is a unit test of the logger serializer, not an integration test
  - Import the httpLogger or the request serializer from logger.ts
  - Call the serializer with a mock request containing url: '/api/secrets/abc123def456ghi789012'
  - Assert the output url is '/api/secrets/[REDACTED]'
```

**Param validation:**
```
test('GET /api/secrets/:id -- rejects invalid ID format with 400')
  - GET /api/secrets/tooshort
  - Expect 400 with error: 'validation_error'
```

**Expiration variants:**
```
test('POST /api/secrets -- accepts all valid expiresIn options')
  - Test each: '1h', '24h', '7d', '30d'
  - All should return 201
```
