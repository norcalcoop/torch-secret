  <name>Task 1: Install node-cron and create expiration worker module</name>
  <files>package.json, package-lock.json, server/src/workers/expiration-worker.ts</files>
  <action>
    1. Install node-cron v4: `npm install node-cron`. No @types/node-cron needed (v4 bundles TypeScript declarations).

    2. Create `server/src/workers/expiration-worker.ts` with two exported functions:

    `startExpirationWorker()`:
    - Uses `cron.schedule('*/5 * * * *', callback)` to run every 5 minutes.
    - The callback performs a two-step bulk cleanup:
      - Step 1: `db.update(secrets).set({ ciphertext: '0' }).where(lte(secrets.expiresAt, now))` -- zero ciphertext for data remanence mitigation (SECR-08). Use single character '0' for batch (not length-matched -- see Research Pitfall 6).
      - Step 2: `db.delete(secrets).where(lte(secrets.expiresAt, now))` -- delete the zeroed rows.
    - Log the deletion count via pino logger: `logger.info({ deletedCount }, 'Expired secrets cleaned up')`. Never log secret IDs (SECR-09).
    - Wrap entire callback in try/catch. On error: `logger.error({ err }, 'Expiration worker error')`. Do NOT re-throw (prevents process crash).
    - Import `lte` from `drizzle-orm`, `db` from `../db/connection.js`, `secrets` from `../db/schema.js`, `logger` from `../middleware/logger.js`.

    `stopExpirationWorker()`:
    - Calls `task.stop()` on the stored cron task reference.
    - Sets the task reference to null.

    Module-level variable: `let task: ReturnType<typeof cron.schedule> | null = null;`

    Note on rowCount: Drizzle's delete returns a QueryResult. Use `.rowCount` if available from the pg driver passthrough. If it's not directly accessible, use `.returning()` with `.length` instead. Test during implementation and pick whichever works.
  </action>

<task type="auto">
  <name>Task 2: Add expiration guards to service layer and integrate worker into server.ts</name>
  <files>server/src/services/secrets.service.ts, server/src/server.ts</files>
  <action>
    **Service layer expiration guards (secrets.service.ts):**

    Add an expiration check to all three functions that retrieve secrets. The check goes AFTER the SELECT and null check, BEFORE any other processing. When a secret is expired, treat it as not found (return null) to maintain the anti-enumeration invariant (SECR-07). Also opportunistically clean up the expired row inline since we already have it in a transaction.

    1. `retrieveAndDestroy()` -- inside the transaction, after `if (!secret) return null;` and BEFORE the password-hash check, add:
       ```
       if (secret.expiresAt <= new Date()) {
         await tx.update(secrets).set({ ciphertext: '0'.repeat(secret.ciphertext.length) }).where(eq(secrets.id, id));
         await tx.delete(secrets).where(eq(secrets.id, id));
         return null;
       }
       ```
       This means expired secrets return null just like nonexistent ones. The caller (route handler) produces the same 404 response.

    2. `getSecretMeta()` -- this function is NOT in a transaction (just a SELECT). Add after `if (!secret) return null;`:
       - Need to also select `expiresAt` in the select clause (currently only selects passwordHash and passwordAttempts).
       - Add `expiresAt: secrets.expiresAt` to the select object.
       - Then check: `if (secret.expiresAt <= new Date()) return null;`
       - No inline cleanup here (no transaction context). The worker or next retrieval attempt will clean it up.

    3. `verifyAndRetrieve()` -- inside the transaction, after `if (!secret) return null;` and BEFORE the passwordHash null check, add:
       ```
       if (secret.expiresAt <= new Date()) {
         await tx.update(secrets).set({ ciphertext: '0'.repeat(secret.ciphertext.length) }).where(eq(secrets.id, id));
         await tx.delete(secrets).where(eq(secrets.id, id));
         return null;
       }
       ```

    **Critical ordering:** The expiration check MUST come before the password-hash check in both `retrieveAndDestroy` and `verifyAndRetrieve`. An expired password-protected secret should return null (not be rejected for having a password).

    **Server.ts integration:**

    Import `startExpirationWorker` and `stopExpirationWorker` from `./workers/expiration-worker.js`.

    - Call `startExpirationWorker()` AFTER `app.listen()` callback (after the server started log).
    - Call `stopExpirationWorker()` at the TOP of the `shutdown()` function, BEFORE `server.close()`. This stops scheduling new cron runs while in-flight work completes.

    The worker is started in server.ts (not in buildApp()) so it does NOT run during integration tests that use buildApp() + supertest.
