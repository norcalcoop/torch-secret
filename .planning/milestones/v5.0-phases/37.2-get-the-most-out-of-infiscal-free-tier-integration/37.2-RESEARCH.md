## Summary

Infisical is an open-source secrets management platform with a CLI-first workflow: `infisical run --env=<env> -- <your-command>` wraps any shell command, fetches secrets from the configured Infisical project/environment, and injects them as environment variables into the child process. No application code changes are required — the app sees standard `process.env` values exactly as it would from a `.env` file.

The free tier is generous for this project: no documented secret count limit, up to 3 projects, 3 environments per project, 5 machine identities, and 10 integrations. Torch Secret needs 1 project, exactly 3 environments (dev/staging/prod), and 2 machine identities (1 for CI, 1 used implicitly by the Render App Connection). The two integrations needed — GitHub Actions and Render secret sync — are both first-class free-tier features. The current `.env.example` lists ~23 keys, comfortably within any reasonable bound.

The CONTEXT.md decision specifies `infisical/secrets-action` with `INFISICAL_TOKEN` from a GitHub Secret. Research confirms Universal Auth (client-id + client-secret) is the correct pattern for that — the secrets-action `method: "universal"` with `client-id` and `client-secret` inputs. OIDC is available as a keyless alternative (no secrets to rotate), but requires additional permissions in the workflow. The planner should resolve whether to use Universal Auth (matches CONTEXT.md wording) or OIDC (preferred best practice, no secret rotation needed). Both are documented below.

**Primary recommendation:** Use `infisical/secrets-action@v1.0.9` with `method: "universal"` and store `INFISICAL_CLIENT_ID` + `INFISICAL_CLIENT_SECRET` as GitHub Secrets (matches CONTEXT.md exactly). For Render, use the native Render App Connection + Secret Sync — no CLI in the Render build command. Local dev and staging use personal `infisical login`.

### Core
| Tool | Version | Purpose | Why Standard |

### Supporting
| Tool | Version | Purpose | When to Use |
|------|---------|---------|-------------|
| `infisical run --env=X --` | CLI subcommand | Wrap any shell command to inject secrets | All dev/staging/CI script wrappers |
| `infisical login` | CLI subcommand | Interactive personal auth for local dev | Developer onboarding (each dev runs once; session persists) |
| `infisical init` | CLI subcommand | Creates `.infisical.json` project config | Run once in repo root to bind to Infisical workspace |

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| Universal Auth for CI (client-id/secret) | OIDC auth (keyless) | OIDC requires no GitHub Secret for auth, never needs rotation; Universal Auth requires storing + rotating `INFISICAL_CLIENT_SECRET`; OIDC is preferred but requires `id-token: write` permission |
| Render native Secret Sync | `infisical run --` in Render build command with INFISICAL_TOKEN | Native sync is simpler; CLI approach requires installing CLI in Render build step |
| Personal `infisical login` for local dev | Shared machine identity token in `.env.local` | Personal login is developer-standard; shared token is an org secret that could be leaked |

### Pattern 1: CLI Wrapper for npm Scripts (dev + staging)

**What:** Prefix every npm script that needs runtime secrets with `infisical run --env=<env> --`. The `--` separator separates Infisical flags from the wrapped command.

**When to use:** `dev:server`, `dev:client`, and new `staging:up` scripts.

**Current scripts and target changes:**
```jsonc
// package.json BEFORE:
"dev:server": "tsx watch server/src/server.ts",
"dev:client": "portless torchsecret sh -c 'npx vite --port $PORT --host $HOST'"

// package.json AFTER:
"dev:server": "infisical run --env=dev -- tsx watch server/src/server.ts",
"dev:client": "infisical run --env=dev -- portless torchsecret sh -c 'npx vite --port $PORT --host $HOST'",
"staging:up": "infisical run --env=staging -- docker-compose up"
```

**How auth works at runtime:**
- Local dev: CLI uses the session from `infisical login` (stored in `~/.infisical/`; runs interactively, no token needed in scripts)
- Staging: Same personal session (developers run `infisical login` once)
- CI: `INFISICAL_TOKEN` env var injected by `secrets-action` or set as a GitHub Secret; CLI auto-detects it

### Pattern 2: .infisical.json Project Binding

**What:** Running `infisical init` in the repo root creates `.infisical.json`. This file binds the CLI to the correct Infisical workspace so `--projectId` is not needed in every script.

**Format:**
```json
// .infisical.json — SAFE TO COMMIT; contains no secrets
// Source: https://infisical.com/docs/cli/project-config (verified)
{
  "workspaceId": "<your-infisical-project-id>",
  "defaultEnvironment": "dev",
  "gitBranchToEnvironmentMapping": null
}
```

**Key facts:**
- Created by `infisical init` — prompts for project selection from your Infisical account
- `workspaceId` is the project UUID shown in Infisical UI project settings
- `defaultEnvironment` sets the fallback env when `--env` is omitted
- File is explicitly safe to commit per Infisical docs ("does not contain any sensitive data")

### Pattern 3: GitHub Actions — infisical/secrets-action (Universal Auth)

**What:** Official action that authenticates with Infisical using machine identity Universal Auth and injects secrets as environment variables for all subsequent steps in the job.

**Auth setup:**
1. In Infisical: Organization Settings > Access Control > Machine Identities > Add Identity → Universal Auth
2. Create a Client Secret — save it (shown once)
3. Store `INFISICAL_CLIENT_ID` and `INFISICAL_CLIENT_SECRET` as GitHub Secrets (not Variables)

**Workflow step:**
```yaml
# Source: https://github.com/Infisical/secrets-action README (verified v1.0.9)
- name: Import secrets from Infisical
  uses: Infisical/secrets-action@v1.0.9
  with:
    method: "universal"
    client-id: ${{ secrets.INFISICAL_CLIENT_ID }}
    client-secret: ${{ secrets.INFISICAL_CLIENT_SECRET }}
    project-slug: "<torch-secret-project-slug>"   # from Infisical project URL/settings
    env-slug: "dev"
    domain: "https://app.infisical.com"
    export-type: "env"
```

Secrets become available as `${{ env.SECRET_NAME }}` in all subsequent steps within the same job.

**CRITICAL: CI infrastructure overrides.** The CI `DATABASE_URL` and `REDIS_URL` point to GitHub Actions service containers (`localhost:5432`, `localhost:6379`) — these must NOT come from Infisical's dev environment. Keep them as job-level `env:` overrides AFTER the secrets-action step:

```yaml
jobs:
  test:
    env:
      # CI-specific: these override any values injected by secrets-action
      DATABASE_URL: postgresql://secureshare:secureshare@localhost:5432/secureshare
      NODE_ENV: test
      BETTER_AUTH_URL: http://localhost:3000
      APP_URL: http://localhost:3000
```

Job-level `env:` declarations in GitHub Actions take precedence over step-level injected vars from the action.

### Pattern 4: GitHub Actions — OIDC Auth (Alternative, Claude's Discretion)

**What:** Keyless OIDC auth — no client secret stored in GitHub. GitHub's OIDC token is used to authenticate with Infisical directly.

**When to use:** If deciding against Universal Auth to avoid secret rotation.

**Requires:** `permissions: id-token: write` on the job or workflow.

```yaml
permissions:
  id-token: write
  contents: read

steps:
  - uses: Infisical/secrets-action@v1.0.9
    with:
      method: "oidc"
      identity-id: ${{ vars.INFISICAL_IDENTITY_ID }}   # non-sensitive; use GitHub Variable not Secret
      project-slug: "torch-secret"
      env-slug: "dev"
      domain: "https://app.infisical.com"
```

**OIDC Machine Identity configuration in Infisical UI:**
```
OIDC Discovery URL: https://token.actions.githubusercontent.com
Issuer:            https://token.actions.githubusercontent.com
Subject (bound):   repo:<owner>/<repo>:*    # glob matches all branches
Audiences:         https://github.com/<owner>
```

### Pattern 5: Render App Connection + Secret Sync

**What:** Infisical holds a Render API key and automatically syncs prod environment secrets to the Render service's environment variables when secrets change.

**Setup steps:**
1. In Render: Account Settings > API Keys > Create API Key (copy immediately — shown once)
2. In Infisical: Project > Integrations > App Connections > Add > Render — paste Render API key, click "Connect to Render"
3. In Infisical: Project > Integrations > Secret Syncs > Add Sync > Render:
   - Source: `prod` environment, path `/`
   - Destination: your Render Service (not Environment Group)
   - Enable Auto-Sync: ON

**Sync behavior:** Secrets sync automatically from Infisical when secrets change. The Render API key (a single Render credential) lives only in Infisical — it is the "bootstrap credential" stored as a Render env var per CONTEXT.md only if the CLI approach is used. For the native sync, Infisical holds the Render API key internally.

**Redeploy consideration:** Render triggers a new deploy when env vars are updated by the sync. Verify this in Plan 03.

### Pattern 6: Local Dev — Personal Login Workflow

**What:** Developers authenticate with their personal Infisical account via browser OAuth. Session persists in `~/.infisical/`.

**Developer onboarding (one time per machine):**
```bash
infisical login              # opens browser, authenticates personal account
cd /path/to/secureshare
# .infisical.json already committed — no infisical init needed
npm run dev:server           # infisical run -- already in the script
npm run dev:client
```

### Pattern 7: Staging Docker Compose Env Var Forwarding (Complication)

**The problem:** Docker Compose `environment:` blocks with literal values (e.g., `BETTER_AUTH_SECRET: 'local-development-secret...'`) take precedence over parent process env vars injected by Infisical. Only bare keys (`KEY:` without value) or `${KEY}` references inherit from the parent process.

**Current docker-compose.yml state:** The `app` service uses literal values. CONTEXT.md says it stays **unchanged**.

**Resolution (Claude's Discretion):** Accept that staging uses docker-compose.yml's existing hardcoded placeholder values for most secrets. The staging Infisical environment's primary purpose is to provide the correct `DATABASE_URL` with Docker internal hostname (`postgresql://user:pass@db:5432/torch`). The `DATABASE_URL` entry in docker-compose.yml is also a literal value (`postgresql://secureshare:secureshare@db:5432/secureshare`) — which is actually fine for local Docker staging since the `db` service uses those credentials. The staging environment may serve as documentation more than active injection.

**Smoke-test to confirm behavior:**
```bash
# Check which DATABASE_URL docker-compose config shows (Infisical vs yaml literal)
infisical run --env=staging -- docker-compose config | grep DATABASE_URL
```

### Anti-Patterns to Avoid

- **Using deprecated service tokens** (`st.xxx`): Infisical deprecated service tokens April 2024. Use machine identities (Universal Auth or OIDC).
- **Not committing `.infisical.json`**: Without it, CI and other developers get "Missing workspace id" errors.
- **Fetching DATABASE_URL from Infisical in CI**: The CI postgres service is at `localhost:5432` with CI-specific credentials. Keep this as a job-level `env:` override.
- **Adding Infisical injection to the `lint` job**: Lint has no env var dependencies. Leave unchanged.
- **Adding Infisical injection to the `docker-build` job**: Docker build passes `VITE_` vars as build args; no runtime secrets needed. Leave unchanged.
- **Using `--command="..."` syntax**: Use `-- <cmd>` (double dash). The `--command` flag is only for chained bash commands with `&&` or `;`.
- **Removing `import 'dotenv/config'` from vitest.config.ts**: It's a harmless no-op when no `.env` exists. Leave it; tests run without a `.env` file because CI gets vars from secrets-action.
- **Putting PostHog personal API key and project ID in Infisical**: These are dev-tooling credentials used only during the Phase 37.1 dashboard setup script. They do not belong in the prod or dev environment.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Secret injection into shell processes | Custom env-loading scripts | `infisical run --` | Handles auth, retry, error reporting, credential refresh automatically |
| GitHub Actions secret injection | Manual `curl` to Infisical API in bash | `Infisical/secrets-action@v1.0.9` | Official action; handles token exchange, exports as env vars for subsequent steps |
| Render env var sync | Script calling Render API + Infisical API | Native Render App Connection + Secret Sync | UI-configured, auto-syncs on change, handles conflicts |
| Bulk secret import | `infisical secrets set KEY=VAL` one-by-one | `infisical secrets set --file=.env` | CLI supports bulk import from `.env` format |

**Key insight:** The integration surface for this phase is exactly three things: (1) prefix npm scripts, (2) add secrets-action step in CI yaml, (3) configure Render sync in Infisical UI. No new code.

## Common Pitfalls

### Pitfall 1: Service Token vs Machine Identity Confusion

**What goes wrong:** Using an old service token (`st.xxx`) instead of a machine identity access token in `INFISICAL_TOKEN`.
**Why it happens:** Service tokens look similar to machine identity tokens and many tutorials still show the old pattern.
**How to avoid:** Create a machine identity with Universal Auth or OIDC. Service tokens are deprecated as of April 2024.
**Warning signs:** Deprecation warnings from Infisical CLI; auth failures after Infisical platform updates.

### Pitfall 2: Missing `.infisical.json` in CI

**What goes wrong:** `infisical run` in CI fails with "Missing environment or workspace id" because `.infisical.json` is not in the checked-out repo.
**Why it happens:** `.infisical.json` was never committed (common if dev created it but `.gitignore` was too broad).
**How to avoid:** Commit `.infisical.json` to git — it contains no secrets.
**Warning signs:** CI step fails immediately with workspace id error.

### Pitfall 3: `--env` Slug Must Match Infisical Slug Exactly

**What goes wrong:** `infisical run --env=Development` fails because the environment slug is `dev`, not `Development`.
**Why it happens:** Infisical uses lowercase slugs, not display names, in the `--env` flag.
**How to avoid:** After creating environments in Infisical UI, check the exact slug in environment settings (usually `dev`, `staging`, `prod`).
**Warning signs:** `Error: Could not find environment with slug "Development"`.

### Pitfall 4: CI DATABASE_URL Collision

**What goes wrong:** The dev environment in Infisical has `DATABASE_URL` pointing to the developer's local Postgres. CI postgres service has different credentials.
**Why it happens:** One Infisical environment is shared between local dev and CI, but the DB endpoint differs.
**How to avoid:** Override `DATABASE_URL` and `REDIS_URL` in ci.yml job-level `env:` block. These overrides take effect after the secrets-action step.
**Warning signs:** CI postgres migration step fails with connection or authentication errors.

### Pitfall 5: Render Sync Does Not Auto-Redeploy (or Does — Verify)

**What goes wrong:** Secrets synced to Render don't take effect until next deploy.
**Why it happens:** Render only applies new env vars on the next deploy.
**How to avoid:** After changing critical secrets, manually trigger a Render deploy. Alternatively, check if Render's sync triggers a redeploy automatically (behavior depends on Render service "auto-deploy" settings).
**Warning signs:** New secret values in Infisical sync shows green, but running app uses old values.

### Pitfall 6: `BETTER_AUTH_TRUSTED_ORIGINS` Missing from `.env.example`

**What goes wrong:** The CONTEXT.md lists `BETTER_AUTH_TRUSTED_ORIGINS` as an app secret, but it is absent from `.env.example`. If not added before migration, Infisical and `.env.example` documentation will be out of sync.
**Why it happens:** Pre-existing gap — `.env.example` was not updated when this variable was added.
**How to avoid:** Add `BETTER_AUTH_TRUSTED_ORIGINS=` entry to `.env.example` during the Plan 02 cleanup pass.
**Warning signs:** App startup auth failures in environments where `BETTER_AUTH_TRUSTED_ORIGINS` is missing.

### Pitfall 7: Docker Compose Literal Values Override Infisical Injection for Staging

**What goes wrong:** `infisical run --env=staging -- docker-compose up` injects secrets at the host process level, but the container gets docker-compose.yml's literal values because those take precedence over host env vars.
**Why it happens:** Docker Compose `environment:` with literal values (`KEY: value`) always wins over inherited process env. Only `KEY:` (bare, no value) or `KEY: ${KEY}` inherits from parent.
**How to avoid:** Per CONTEXT.md, docker-compose.yml stays unchanged. Accept that staging Docker containers use the compose file's hardcoded placeholders for most values. The staging Infisical environment primarily ensures the correct DATABASE_URL with Docker-internal hostname is set — but since docker-compose.yml also hardcodes DATABASE_URL with the Docker `db:5432` hostname, staging Infisical injection may be redundant for Docker. The smoke-test verifies actual behavior.
**Warning signs:** Containers use old placeholder values despite Infisical secrets being updated.

### Pitfall 8: Free Tier 3-Environment Limit

**What goes wrong:** Adding a 4th environment (e.g., `test`) hits the free tier limit.
**Why it happens:** Free tier allows max 3 environments per project.
**How to avoid:** Use exactly dev, staging, prod. CI uses dev environment. No separate test environment.
**Warning signs:** Infisical UI error when creating 4th environment.

### Pitfall 9: `VITE_*` Vars Require Build-Time Injection

**What goes wrong:** `VITE_POSTHOG_KEY` is missing from the built frontend because it was set as a Render env var (runtime) but wasn't available during `vite build` (build time).
**Why it happens:** Vite replaces `import.meta.env.VITE_*` at build time — the value must be in `process.env` when `vite build` runs.
**How to avoid:** Render service env vars (synced from Infisical prod) are available during the Render build phase. Include `VITE_POSTHOG_KEY` and `VITE_POSTHOG_HOST` in the Infisical prod environment and in the Render Secret Sync.
**Warning signs:** Built frontend has `undefined` for PostHog key; no analytics events fire.

### Pitfall 10: CI e2e Job Missing `LOOPS_API_KEY`

**What goes wrong:** Current ci.yml e2e job has 11 env vars (missing `LOOPS_API_KEY` present in test job). E2e tests may fail on Loops-related env validation.
**Why it happens:** Pre-existing omission from Phase 37.
**How to avoid:** When replacing the e2e job env block with Infisical injection, this gap is automatically fixed — Infisical dev environment includes `LOOPS_API_KEY`.
**Warning signs:** E2e job fails with Zod validation error on `LOOPS_API_KEY`.

## Open Questions

1. **Whether job-level `env:` in GitHub Actions overrides secrets-action injected vars**
   - What we know: GitHub Actions job-level `env:` declarations are set before steps run; they should take precedence over step-level injected env vars
   - What's unclear: The exact precedence order between job-level `env:` and `secrets-action`-injected vars; GitHub's documentation on this is not perfectly clear
   - Recommendation: Place CI-specific overrides (DATABASE_URL, NODE_ENV) in the job-level `env:` block. Verify during Plan 03 smoke test that CI postgres connection works.

2. **Does Render Secret Sync trigger an automatic service redeploy?**
   - What we know: Render deploys on env var changes if "auto-deploy" is enabled on the service; the sync updates stored env vars
   - What's unclear: Whether Infisical's Render API sync call reliably triggers Render's change detection
   - Recommendation: Test with a non-critical secret change (e.g., LOG_LEVEL) during Plan 03 verification.

3. **Docker Compose staging injection utility**
   - What we know: docker-compose.yml has literal values that take precedence over host process env vars; CONTEXT.md says docker-compose.yml stays unchanged
   - What's unclear: Whether the staging Infisical environment provides any practical value if docker-compose.yml's literal values always win
   - Recommendation: The staging env provides the correct DATABASE_URL with `db:5432` hostname. Since docker-compose.yml already has `DATABASE_URL: postgresql://secureshare:secureshare@db:5432/secureshare` hardcoded, the staging injection is effectively a no-op. Plan 03 smoke-test will clarify. The staging environment may primarily serve as documentation and a fallback if docker-compose.yml is later updated.

4. **Authentication method for CI: Universal Auth vs OIDC**
   - What we know: CONTEXT.md says "Authenticates using `INFISICAL_TOKEN` from GitHub Secrets" — this maps to Universal Auth pattern (or a pre-generated access token). The secrets-action supports both Universal Auth and OIDC.
   - Recommendation (Claude's Discretion): Use Universal Auth to match CONTEXT.md exactly (store INFISICAL_CLIENT_ID + INFISICAL_CLIENT_SECRET as GitHub Secrets). OIDC would be preferable long-term but requires scope changes.
