**Event coverage gaps**
- Pricing page: no additional events needed — existing `capturePageview()` already fires on route change; pricing page visits appear in the funnel automatically.

  - `checkout_initiated` — fired client-side when the user clicks the upgrade CTA and is redirected to Stripe Checkout

- Anonymous users: no changes — PostHog auto-assigns an anonymous distinct_id; anonymous events aggregate normally.

## Summary

Phase 37.1 enriches an already-functioning PostHog integration. The existing `posthog.ts` module (posthog-js 1.352.0) handles initialization, fragment sanitization, pageviews, and seven analytics functions. This phase adds three new event-capture functions, extends two existing ones, and adds a person-properties layer — all entirely client-side in `client/src/analytics/posthog.ts`, with call sites added to `dashboard.ts` and `create.ts`.

The zero-knowledge invariant constrains every new event: `checkout_initiated` and `subscription_activated` must carry no `secretId`; the new `protection_type` property on `secret_created` must be derived from client-side UI state, never from the server response. The `identifyUser()` extension adds `setPersonProperties()` for tier and registration date — both non-PII. INVARIANTS.md must be updated before any code change, as per the mandatory protocol.

The PostHog free tier is generous for a launch-stage product: 1 million events/month, unlimited dashboards, unlimited funnels, unlimited cohorts — all plan-unrestricted features. No SDK changes are needed for the PostHog UI configuration work (dashboards, funnels); those are manual operations in the PostHog cloud interface.

**Primary recommendation:** Implement all changes in `posthog.ts` first (module API), then thread calls through `dashboard.ts` and `create.ts`. Update INVARIANTS.md before writing any code.

## Architecture Patterns

### Recommended Project Structure

All analytics changes live in one module. Call sites thread through page modules:

```
client/src/
├── analytics/
│   └── posthog.ts              # All new functions added here
├── pages/
│   ├── create.ts               # captureSecretCreated() gains protection_type param
│   └── dashboard.ts            # identifyUser() extended; 3 new capture calls added
└── .planning/INVARIANTS.md     # MUST be updated first — extension protocol
```

### Pattern 1: Module-API First

**What:** Add all new exported functions to `posthog.ts` before modifying call sites. This keeps analytics logic centralized and testable independently.

**When to use:** Always — the module is the single source of truth for PostHog calls.

**Example:**
```typescript
// Source: client/src/analytics/posthog.ts — existing pattern to follow

/**
 * Capture checkout_initiated event when upgrade CTA is clicked.
 *
 * ZERO-KNOWLEDGE INVARIANT: no secretId, no userId in event payload.
 * source: which CTA triggered the upgrade (pricing_page or conversion_prompt)
 */
export function captureCheckoutInitiated(source: 'pricing_page' | 'conversion_prompt'): void {
  if (!isInitialized()) return;
  posthog.capture('checkout_initiated', { source });
}
```

### Pattern 2: Person Properties on Identify

**What:** Extend `identifyUser()` to call `posthog.setPersonProperties()` immediately after `posthog.identify()`. The SDK sends a `$set` event that updates the person profile.

**When to use:** On every successful login/dashboard load that calls `identifyUser()`.

**Example:**
```typescript
// Source: posthog-js 1.352.0 lib/src/posthog-core.d.ts (verified from installed package)

// BEFORE (current):
export function identifyUser(userId: string): void {
  if (!isInitialized()) return;
  posthog.identify(userId);
}

// AFTER (this phase):
export function identifyUser(userId: string, tier: 'free' | 'pro', registeredAt: string): void {
  if (!isInitialized()) return;
  posthog.identify(userId);
  posthog.setPersonProperties({ tier, registered_at: registeredAt });
}
```

**API signatures verified from installed package (`lib/src/posthog-core.d.ts`):**
```typescript
identify(new_distinct_id?: string, userPropertiesToSet?: Properties, userPropertiesToSetOnce?: Properties): void;
setPersonProperties(userPropertiesToSet?: Properties, userPropertiesToSetOnce?: Properties): void;
```

**What:** After `subscription_activated` fires, call `posthog.setPersonProperties({ tier: 'pro' })` as a standalone property update — no full re-identify needed.

**When to use:** The success redirect page already has the session; just update the tier without re-calling `identify()`.

**Example:**

### Pattern 4: Deriving protection_type from UI State

**What:** `getProtectionType()` in `create.ts` currently returns `'none' | 'passphrase' | 'password'` — collapsing both `generate` and `custom` tabs into `'password'`. The CONTEXT.md requires four distinct values matching the four UI tabs.

**When to use:** The `captureSecretCreated()` call in `create.ts` already has access to `protectionType` from `protectionPanel.getProtectionType()`. Two options:

**Option A (preferred — CONTEXT.md aligned):** Change `captureSecretCreated` to accept a `protection_type: 'none' | 'passphrase' | 'password' | 'generated'` parameter, and change the call site to map the active tab directly:

```typescript
// In create.ts submit handler — line ~1254
const rawType = protectionPanel.getProtectionType(); // 'none' | 'passphrase' | 'password'
// Distinguish generate vs custom via panel's activeTab — but getProtectionType() collapses them.
// Simpler: expose getActiveTab() OR pass the tab ID directly to capture.
```

**Conflict:** `getProtectionType()` cannot distinguish `generate` from `custom` — both return `'password'`. To emit `'generated'` vs `'password'` for the analytics event, the call site needs the tab ID directly. Options:
1. Add `getActiveTabId(): 'none' | 'generate' | 'custom' | 'passphrase'` export from `createProtectionPanel()`
2. Map in the capture call site: if `activeTab === 'generate'` → `'generated'`; if `activeTab === 'custom'` → `'password'`

Option 1 is cleaner and avoids leaking tab state to the submit handler. The planner should decide between these — either is valid.

### Pattern 5: dashboard_viewed Event

**What:** Fire `dashboard_viewed` once per dashboard page load, after `identifyUser()` resolves.

**When to use:** In `renderDashboardPage()`, after the `identifyUser()` call — not gated on secrets count.

```typescript
// After identifyUser() call in dashboard.ts
captureDashboardViewed();
```

```typescript
// In posthog.ts
export function captureDashboardViewed(): void {
  if (!isInitialized()) return;

```

### Anti-Patterns to Avoid

- **Re-identify on every event:** Call `posthog.identify()` once per session (on login/dashboard load). Never call it inside event handlers like button clicks.
- **PII in person properties:** Never pass email, name, or display name to `setPersonProperties()`. `tier` and `registered_at` are the only allowed properties per locked decision.
- **secretId in upgrade events:** `checkout_initiated` and `subscription_activated` must carry no `secretId`. These events fire in billing/upgrade flows that have no access to any secret ID.
- **Calling setPersonProperties before identify:** For new users, `identify()` must run before `setPersonProperties()` — otherwise the properties are set on the anonymous profile, not the identified one. In the current codebase, `identifyUser()` already calls `posthog.identify(userId)` first, so the extension is safe.

## Common Pitfalls

### Pitfall 1: identifyUser() Signature Change Breaks Existing Callers

**What goes wrong:** `identifyUser(userId)` is called from `login.ts` (line 215) and `dashboard.ts` (line 386). Adding required parameters to the signature breaks both callers.

**Why it happens:** TypeScript will catch this at compile time, but the fix requires updating every call site.

**How to avoid:** Make the new parameters optional with defaults, or pass an options object:
```typescript
export function identifyUser(userId: string, tier?: 'free' | 'pro', registeredAt?: string): void
```
Alternatively, keep `identifyUser(userId)` unchanged and add a separate `updatePersonProperties(tier, registeredAt)` function called explicitly at each site where the data is available.

**Warning signs:** TypeScript compile error on any callers not updated; missing person properties in PostHog for login-path users.

**Research note:** `login.ts` at line 215 calls `identifyUser(user['id'])`. It does NOT have access to `tier` or `registeredAt` — those require a `/api/me` call. Making the extended parameters optional is the correct approach for the login path.

### Pitfall 2: subscription_activated Fires on Every Dashboard Load

**What goes wrong:** The `isUpgraded` + `checkoutSessionId` check in `dashboard.ts` (lines 375-376) already gates the success banner. If `captureSubscriptionActivated` is not gated by the same condition, it fires on every dashboard visit.

**Why it happens:** The analytics call must be co-located with the `verifyCheckoutSession()` success branch, not at the top of `renderDashboardPage()`.

**How to avoid:** Place `captureSubscriptionActivated()` inside the `isUpgraded && checkoutSessionId` guard block, after `verifyCheckoutSession()` resolves successfully (on the success branch, not the catch branch).

### Pitfall 3: checkout_initiated vs subscription_activated Source Confusion

**What goes wrong:** The `checkout_initiated` event fires from two locations — the pricing page Pro CTA (navigates to `/register?plan=pro`) and the dashboard upgrade button. The CONTEXT.md specifies a `source` property: `'pricing_page' | 'conversion_prompt'`.

**Why it happens:** The pricing page CTA goes to `/register?plan=pro`, not to Stripe Checkout directly. Looking at `pricing.ts`, the Pro card CTA navigates to `/register?plan=pro` — it does NOT call `initiateCheckout()`. The `initiateCheckout()` call is in `dashboard.ts`. The event should fire right before `window.location.href = url` in the dashboard upgrade button handler.

**How to avoid:** The `checkout_initiated` event fires only when `initiateCheckout()` is called (dashboard upgrade button). Source is `'pricing_page'` only if the upgrade was triggered from a pricing page CTA — but the pricing page CTA goes to register, not checkout. So for launch: source is always `'dashboard'` for the dashboard upgrade button. The CONTEXT.md also mentions `'conversion_prompt'` as a source — this would be for a future pricing-page-to-checkout path.

**Research note (MEDIUM confidence):** Re-reading CONTEXT.md: `checkout_initiated` should carry `{ source: 'pricing_page' | 'conversion_prompt' }`. Currently, only the dashboard has an `initiateCheckout()` call. If the pricing page ever adds a direct-to-Stripe CTA (bypassing register), the source distinction will matter. For now: source is always derived from where the upgrade button lives — dashboard upgrade button = one source value.

### Pitfall 4: registered_at Data Availability

**What goes wrong:** `identifyUser()` is called in `dashboard.ts` after session check (line 386). The session object has `session.user.id` and `session.user.email` but NOT `registered_at` or `subscriptionTier`. Those require a `/api/me` call.

**Why it happens:** Better Auth's session object does not include custom columns from the users table.

**How to avoid:** The `/api/me` call is already made in `dashboard.ts` (lines 363-370) to get `subscriptionTier`. Extend the `MeResponse` type in `shared/types/api.ts` to also return `createdAt` (ISO date string), then use it when calling `identifyUser()`. The `getMe()` call is already there — it just needs the additional field.

**Warning signs:** `registered_at` shows as undefined in PostHog person profiles if the ME response doesn't include it.

### Pitfall 5: INVARIANTS.md Update Skipped

**What goes wrong:** New analytics events are written without updating INVARIANTS.md first, violating the mandatory protocol.

**Why it happens:** It's easy to jump straight to code.

**How to avoid:** Per `CLAUDE.md`: "Before writing any code that touches the database schema, application logger, or analytics events, read `.planning/INVARIANTS.md`." The first task in Wave 0 must be updating INVARIANTS.md with the new events and their invariant compliance notes.

## Open Questions

1. **protection_type 4-value mapping: expose activeTab or map in capture call**
   - What we know: `getProtectionType()` returns 3 values (`'none'`, `'passphrase'`, `'password'`). CONTEXT.md requires 4 (`'none'`, `'passphrase'`, `'password'`, `'generated'`). The `generate` and `custom` tabs both return `'password'` from the current function.
   - What's unclear: Should `createProtectionPanel()` add `getActiveTabId()` export, or should the submit handler inline the mapping?
   - Recommendation: Add `getActiveTabId(): 'none' | 'generate' | 'custom' | 'passphrase'` to the panel return object. The submit handler then maps: `generate → 'generated'`, `custom → 'password'`, `passphrase → 'passphrase'`, `none → 'none'`. Keeps analytics mapping logic close to the capture call.

2. **registered_at data path: requires /api/me extension**
   - What we know: `dashboard.ts` already calls `getMe()` for `subscriptionTier`. `registered_at` would require adding `createdAt` to the ME response.
   - What's unclear: Whether the planner should treat ME endpoint extension as a separate task or fold it into the `identifyUser` extension task.
   - Recommendation: One task covers both — extend ME response + extend `identifyUser()`. Small scope addition.

3. **checkout_initiated source values**
   - What we know: Currently only the dashboard has an `initiateCheckout()` call. CONTEXT.md mentions `'pricing_page' | 'conversion_prompt'` as source values.
   - What's unclear: Whether a pricing page "Get Pro" CTA should eventually bypass register and go straight to Stripe.
   - Recommendation: For this phase, implement source as a parameter; the dashboard button passes `'dashboard'`. Add `'pricing_page'` and `'conversion_prompt'` as valid values for future use. This avoids a breaking API change later.
