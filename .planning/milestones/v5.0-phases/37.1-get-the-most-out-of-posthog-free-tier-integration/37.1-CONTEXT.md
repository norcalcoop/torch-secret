# Phase 37.1: Get the Most Out of PostHog Free Tier Integration - Context

**Gathered:** 2026-02-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Maximize value from the existing PostHog free tier integration — both through code-side event improvements (new events, richer event properties, user property updates) and PostHog-side configuration guidance (dashboards, funnels, cohorts). No new analytics providers. No session recording. No changes to the zero-knowledge invariants.

</domain>

<decisions>
## Implementation Decisions

### Event coverage gaps

- Pricing page: no additional events needed — existing `capturePageview()` already fires on route change; pricing page visits appear in the funnel automatically.
- Stripe upgrade flow: add two new events:
  - `checkout_initiated` — fired client-side when the user clicks the upgrade CTA and is redirected to Stripe Checkout
  - `subscription_activated` — fired client-side on the Stripe success redirect page (when `?session_id=` query param is present), confirming a completed Pro upgrade
- `secret_created` event: add `protection_type: 'none' | 'passphrase' | 'password' | 'generated'` property alongside existing `expires_in` and `has_password`. Maps directly to the four protection panel tabs on the create page.
- Dashboard tracking: Claude's discretion (see below).

### User properties & cohorts

- `identifyUser()` should be extended to also call `posthog.setPersonProperties()` with: `{ tier: 'free' | 'pro', registered_at: <ISO date string> }` — no PII (no email, no name).
- On upgrade (`subscription_activated`), immediately call `posthog.setPersonProperties({ tier: 'pro' })` to update the person profile without waiting for next login.
- The `tier` update fires client-side on the Stripe success redirect, not server-side — no posthog-node SDK required.
- Anonymous users: no changes — PostHog auto-assigns an anonymous distinct_id; anonymous events aggregate normally.

### Funnels & dashboards (PostHog UI configuration)

- Primary conversion funnel: `$pageview` (/) → `secret_created` → `user_registered` → `subscription_activated`. This is the main free-to-paid funnel and the primary drop-off signal.
- Conversion prompt funnel: `conversion_prompt_shown` → `conversion_prompt_clicked` → `user_registered`. Measures upsell nudge effectiveness.
- Launch dashboard widgets: DAU (daily active users), secrets created per day, registrations per day, Pro upgrades per day. These four numbers answer "is anyone using it and are they converting?"
- SEO/traffic split: Claude's discretion (see below).

### Feature flags

- No feature flags for launch — shipping to 100% of users.
- No A/B tests at launch — baseline traffic needed first before experiments are statistically meaningful.
- No PostHog Surveys — Tally.so feedback links (Phase 38) handle feedback collection; a parallel PostHog survey channel adds overhead without additive value.

### Claude's Discretion

- Dashboard tracking: add `dashboard_viewed` event only (no individual secret-management actions). A lightweight signal that identifies engaged power users without tracking internal management operations.
- SEO/traffic split on launch dashboard: add a "Top pages" widget showing pageview counts split by route (`/`, `/create`, `/pricing`, `/dashboard`) — gives an early signal on which entry point drives more engagement without making the dashboard cluttered.
- PostHog Surveys: skip entirely for this phase. Phase 38's Tally.so feedback link is the chosen feedback mechanism.

</decisions>

<specifics>
## Specific Ideas

- The `subscription_activated` event fires on the Stripe success redirect page — the existing checkout success handling in the SPA already loads after `?session_id=` redirect. That's where both the event and the tier property update should fire.
- Zero-knowledge invariant applies to all new events: `checkout_initiated` and `subscription_activated` must carry no `secretId`. `checkout_initiated` can carry `{ source: 'pricing_page' | 'conversion_prompt' }` to distinguish where upgrades originate.
- `protection_type` on `secret_created` must be inferred from the UI state, not from the server response — the client already knows which tab was active.

</specifics>
