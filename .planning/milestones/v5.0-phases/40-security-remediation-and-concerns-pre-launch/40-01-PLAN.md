<objective>
Harden POST /api/secrets/:id/verify against brute-force and DoS attacks by layering two defenses: a tight 5 req/min rate limiter and a max-4 concurrency cap on Argon2id. Enforce a 100KB payload ceiling on POST /api/secrets at both the Zod schema and error-handler layers. Tighten the E2E bypass safety gate and add structured logging for Redis store failures.

Purpose: Items #1 (tight rate limit) and #2 (Argon2id concurrency) from the security posture report directly protect against CPU-exhaustion DoS — an attacker who bypasses the rate limit via distributed IPs can still hammer Argon2id. Item #6 (payload limit) prevents memory-exhaustion attacks. CONCERNS.md quick wins (E2E gate, passOnStoreError warn) are co-located in the same files modified here.

Output: Updated rate-limit.ts (new limiter + passOnStoreError warn + E2E gate hardening), password.service.ts (p-limit wrap), secrets.ts (limiter wiring), shared/types/api.ts (Zod max), error-handler.ts (413 + 503 handlers).
</objective>

     * Wraps a rate-limit Store so that any increment/decrement error is logged
     * as a Pino warn before re-throwing. The re-throw allows passOnStoreError to
     * pass the request through, while the warn gives observability.

    function wrapStoreWithWarnOnError(store: Store | undefined): Store | undefined {
      if (!store) return undefined;
      const original = store.increment.bind(store);
      store.increment = async (...args) => {
        try {
          return await original(...args);
        } catch (err) {

          throw err;
        }
      };
      return store;
    }
