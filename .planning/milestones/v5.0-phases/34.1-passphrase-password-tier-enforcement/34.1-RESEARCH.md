## Summary

Phase 34.1 gates the four protection modes on the create page by account tier. The current implementation uses a horizontal ARIA tablist (`role=tablist`/`role=tab`/`role=tabpanel`) inside `createProtectionPanel()` in `client/src/pages/create.ts`. The phase decision is to keep all options always visible but add per-tier lock states — which means the tab strip approach must be redesigned. The locked tabs need a different interaction model (click opens popover, not panel) rather than the standard tab activation. The CONTEXT.md decision to "extend the existing ARIA combobox component (same pattern as the expiration combobox)" is actually a design direction reference — the current protection panel is a tablist, not a combobox. The correct reading is: apply the same gating pattern (visible-but-locked rows with popover) to the protection panel's tab strip.

The API side is straightforward: add `protection_type` enum to `CreateSecretSchema` in `shared/types/api.ts`, then enforce it inside the POST `/api/secrets` route handler, using the same guard pattern as the existing `expiresIn` tier cap (lines 83–99 of `server/src/routes/secrets.ts`). The `optionalAuth` middleware already populates `res.locals.user` with the user's session including tier information accessible via a `getMe()` call pattern, but the route handler already has `user` from `res.locals.user`. The `subscriptionTier` lives on the users table and is returned by `auth.api.getSession()` only if explicitly queried — the route handler must look it up from the DB or use the pattern from `/api/me`.

**Primary recommendation:** Refactor `createProtectionPanel()` to accept `{ isAuthenticated, isPro }` params and convert locked tabs to non-activatable items with inline popover CTAs; add `protection_type` to `CreateSecretSchema` and enforce tier caps in the server route handler with the same pattern as the existing `expiresIn` cap.

### Pattern 1: Parameterized Protection Panel

The existing `createProtectionPanel()` takes no parameters. It must be refactored to accept tier information so it can gate tabs at render time.

**What:** Convert `createProtectionPanel()` to `createProtectionPanel({ isAuthenticated, isPro })`. Tabs that the caller's tier cannot use render as locked (no activation, popover on click).

**When to use:** Called from the auth IIFE in `renderCreatePage()`, same place `createExpirationSelect(isAuthenticated, isPro)` is already called.

```typescript
// Source: inferred from existing create.ts auth IIFE pattern (lines 1186–1225)

// BEFORE (current — no tier awareness):
const protectionPanel = createProtectionPanel();
form.appendChild(protectionPanel.element);

// AFTER (Phase 34.1):
// Initial render (anonymous): protection panel created with anonymous state
let protectionPanel = createProtectionPanel({ isAuthenticated: false, isPro: false });
form.appendChild(protectionPanel.element);

// Then in the auth IIFE, after isPro is resolved:
// Remove and replace, same as expiration select upgrade pattern
protectionPanel.element.remove();
protectionPanel = createProtectionPanel({ isAuthenticated: true, isPro });
form.insertBefore(protectionPanel.element, errorArea);
```

**Lock states by tier:**

| Tab | Anonymous | Free | Pro |
|-----|-----------|------|-----|
| No protection | Unlocked (default selected) | Unlocked | Unlocked |
| Generate passphrase | "Free" lock → /register | Unlocked | Unlocked |
| Set password (custom) | "Pro" lock → /pricing | "Pro" lock → /pricing | Unlocked |
| Custom password | "Pro" lock → /pricing | "Pro" lock → /pricing | Unlocked |

Note: The CONTEXT.md decision collapses "Generate password" and "Custom password" into distinct "Pro"-locked variants for anonymous/free users. Looking at the existing tab definitions (`tabDefs`):
- `'none'` = No protection (always available)
- `'generate'` = Generate password (Pro only for anonymous users per decision; Free gets passphrase but not this)
- `'custom'` = Custom password (Pro only)
- `'passphrase'` = Passphrase (Free+, locked for anonymous)

The decision text says: "Free: generated passphrase only" — meaning the `'passphrase'` tab is the passphrase tab, and `'generate'`/`'custom'` are password tabs. Anonymous: only `'none'` usable.

### Pattern 2: Locked Tab with Inline Popover

Locked tabs are real `role=tab` elements that are not activatable. Clicking them opens an anchored popover instead.

**What:** A tab button that does not call `activateTab()` on click. Instead shows/hides a popover div anchored below the tab.

**Implementation strategy:**
- Tab button gets `aria-disabled="true"` and a badge span ("Free" or "Pro")
- Click handler opens popover (positioned absolute, anchored to the tab button)
- Popover contains: feature description + CTA link (`navigate('/register')` or `navigate('/pricing')`)
- Clicking outside or pressing Escape closes popover
- Active selection does not change

```typescript
// Source: based on existing lock pattern in expiration-select.ts (lines 146–183)
// and existing navigate() from router.ts

function buildLockedTab(
  id: ActiveTab,
  label: string,
  lockLevel: 'free' | 'pro',
  currentSelectionRef: { value: ActiveTab },
): HTMLButtonElement {
  const btn = document.createElement('button');
  btn.type = 'button';
  btn.role = 'tab';
  btn.id = `tab-btn-${id}`;
  btn.setAttribute('aria-disabled', 'true');
  btn.setAttribute('aria-selected', 'false');
  btn.className = tabInactiveCls + ' relative'; // add relative for popover anchoring

  const labelSpan = document.createElement('span');
  labelSpan.textContent = label;
  btn.appendChild(labelSpan);

  const badge = document.createElement('span');
  badge.className = 'ml-1 text-[10px] font-semibold px-1 py-0.5 rounded bg-surface-raised text-text-muted';
  badge.textContent = lockLevel === 'free' ? 'Free' : 'Pro';
  btn.appendChild(badge);

  // Popover (inline, anchored to this tab)
  const popover = document.createElement('div');
  popover.hidden = true;
  popover.className =
    'absolute top-full left-0 z-50 mt-1 w-56 rounded-lg border border-border ' +
    'bg-surface shadow-lg p-3 space-y-2 text-left';
  popover.setAttribute('role', 'tooltip');

  const description = document.createElement('p');
  description.className = 'text-xs text-text-secondary';
  description.textContent = lockLevel === 'free'
    ? 'Create a free account to use generated passphrases.'
    : 'Upgrade to Pro to set custom passwords.';
  popover.appendChild(description);

  const ctaLink = document.createElement('a');
  ctaLink.href = lockLevel === 'free' ? '/register' : '/pricing';
  ctaLink.className = 'text-xs font-medium text-accent hover:underline';
  ctaLink.textContent = lockLevel === 'free' ? 'Sign up free →' : 'Upgrade →';
  ctaLink.addEventListener('click', (e) => {
    e.preventDefault();
    navigate(lockLevel === 'free' ? '/register' : '/pricing');
  });
  popover.appendChild(ctaLink);

  btn.appendChild(popover);

  btn.addEventListener('click', (e) => {
    e.stopPropagation();
    popover.hidden = !popover.hidden;
  });

  return btn;
}
```

### Pattern 3: API `protection_type` Enforcement

Mirror the existing `expiresIn` tier cap pattern in `server/src/routes/secrets.ts` (lines 82–99). The handler already has `userId` from `optionalAuth` but does NOT have `subscriptionTier` on the `AuthUser` object from Better Auth session. The route must look up tier from the DB.

**What:** Add `protection_type` to `CreateSecretSchema`, then in the route handler, validate that the caller's tier allows the requested protection type.

**Getting subscriptionTier in the route handler:**

The `res.locals.user` is the Better Auth `session.user` object. Based on how `GET /api/me` works (it does a separate DB lookup via Drizzle to get `subscriptionTier` — see `server/src/routes/me.ts`), the secrets route must do the same when a non-none protection type is submitted.

```typescript
// Source: same pattern as expiresIn cap in secrets.ts (lines 82–99)

// In POST /api/secrets handler, after existing expiresIn checks:
const protectionType = body.protection_type ?? 'none';

if (protectionType !== 'none') {
  if (!userId) {
    // Anonymous: no protection allowed
    res.status(403).json({
      error: 'passphrase_not_allowed',
      message: 'Password protection requires a free account. Sign up to use passphrases.',
    });
    return;
  }

  if (protectionType === 'password') {
    // Password requires Pro — look up tier
    const userRow = await db.select({ subscriptionTier: users.subscriptionTier })
      .from(users)
      .where(eq(users.id, userId))
      .limit(1);
    const tier = userRow[0]?.subscriptionTier ?? 'free';
    if (tier !== 'pro') {
      res.status(403).json({
        error: 'pro_required',
        message: 'Custom password protection is a Pro feature. Upgrade to use it.',
      });
      return;
    }
  }
  // protectionType === 'passphrase': authenticated (any tier) is allowed
}
```

### Pattern 4: `protection_type` in Shared Schema

```typescript
// Source: shared/types/api.ts — extend CreateSecretSchema

export const CreateSecretSchema = z.object({
  ciphertext: z.string().min(1).max(200_000),
  expiresIn: z.enum(['1h', '24h', '7d', '30d']),
  password: z.string().min(1).max(128).optional(),
  label: z.string().max(100).optional(),
  notify: z.boolean().optional(),
  /** Explicit protection type for tier enforcement (Phase 34.1) */
  protection_type: z.enum(['none', 'passphrase', 'password']).optional().default('none'),
});
```

The `password` field remains optional and is the actual credential. `protection_type` is the tier gate signal. The server validates: if `protection_type === 'password'`, caller must be Pro. If `protection_type === 'passphrase'`, caller must be authenticated (any tier). If `protection_type === 'none'`, no restriction.

**Note:** The passphrase tab in the existing implementation does NOT send any password to the API (see `getPassword()` in create.ts lines 919–933 — passphrase tab returns `undefined`). So `protection_type: 'passphrase'` is a signal-only field for tier validation; no `password` value accompanies it. The server must accept a passphrase request where `password` is absent but `protection_type === 'passphrase'` is set.

### Anti-Patterns to Avoid

- **Hiding locked options instead of showing them locked:** The decision is explicit — all options always visible. Hidden options violate the UX requirement.
- **Using `aria-selected="true"` on locked tabs:** Locked tabs must never become the selected tab. Keep `aria-selected="false"` permanently on locked items.
- **Re-creating the entire protection panel on auth resolve:** Use the replace pattern (`.remove()` + re-insert) same as `createExpirationSelect` upgrade. Do not try to patch the existing panel in-place — too many closures to update safely.
- **Inferring protection type from `password` field presence:** The CONTEXT.md decision is to use an explicit `protection_type` enum. Server must use `protection_type`, not `!!body.password`.
- **DB tier lookup on every request:** Only look up `subscriptionTier` when `protection_type !== 'none'` (i.e., when a tier check is actually needed). Anonymous requests with `protection_type: 'none'` should never hit the DB for tier data.
- **Letting the submit handler fire when a locked tab is "active":** The lock must prevent activation — `activeTab` closure must never be set to a locked tab id.

## Common Pitfalls

### Pitfall 1: `activeTab` closure captures old values after panel replace

**What goes wrong:** If the protection panel is replaced (`.remove()` + re-insert) in the auth IIFE, any closures in the old panel (e.g., `getPassword()`, `getPassphrase()`) still reference the old panel's `activeTab`. The `form.addEventListener('submit')` captures `protectionPanel` by reference — if replaced correctly this works, but requires `let protectionPanel = ...` not `const`.

**Why it happens:** The submit handler captures `protectionPanel` by reference. If the variable is reassigned, the new panel's accessors are used on next submit. This is correct behavior IF the variable is declared with `let`.

**How to avoid:** Declare `let protectionPanel = createProtectionPanel(...)` (not `const`). The submit handler reads `protectionPanel.getPassword()` and `protectionPanel.getPassphrase()` at call time, so it will use the updated panel after replacement.

**Warning signs:** Submit handler still returns `undefined` from `getPassword()` after tier upgrade.

### Pitfall 2: Tab `activeTab` state set to locked tab on keyboard nav

**What goes wrong:** The existing `tabList.addEventListener('keydown')` wraps through all tabs with ArrowRight/ArrowLeft. After locking some tabs, keyboard nav must skip locked tabs.

**Why it happens:** The `tabOrder` array in the keydown handler includes all four tab IDs. ArrowRight from an unlocked tab will land on a locked tab and call `activateTab()`.

**How to avoid:** Filter `tabOrder` to only include unlocked tabs in the keyboard nav handler. Locked tabs should be skipped entirely.

**Warning signs:** ArrowRight from "No protection" activates "Passphrase" (which should be locked for anonymous) and renders its panel.

### Pitfall 3: `protection_type: 'passphrase'` on re-render triggers API 403 for anonymous users

**What goes wrong:** If an anonymous user somehow has `protection_type: 'passphrase'` in the request body (e.g., due to a client-side bug), the server should reject with 403. The client must never send non-'none' protection_type for anonymous users.

**Why it happens:** The client-side lock prevents selection, but defense-in-depth requires server enforcement.

**How to avoid:** Server validates `protection_type` against tier independently. Client is UX; server is authority.

### Pitfall 4: NodeListOf iteration in tab navigation

**What goes wrong:** `Array.from()` is required when iterating `NodeListOf<HTMLButtonElement>` under `ES2022` lib target (as noted in Phase 34 execution notes in STATE.md).

**Why it happens:** `tsconfig` does not include downlevel iterator support for DOM NodeList.

**How to avoid:** When querying tab buttons for keyboard navigation, use `Array.from(tabList.querySelectorAll<HTMLButtonElement>('[role=tab]:not([aria-disabled])'))`.

### Pitfall 5: `protection_type` default missing from existing API clients

**What goes wrong:** Old API calls from `client/src/api/client.ts` that don't send `protection_type` will be treated as `'none'` (because `.optional().default('none')` in Zod handles missing field). This is correct default behavior.

**How to avoid:** No migration needed; Zod default handles omitted field gracefully.

## Open Questions

1. **Where does "Generate passphrase" land in the tier matrix?**
   - What we know: CONTEXT.md says "Free: generated passphrase only" and the existing tablist has `'generate'` (Generate password), `'custom'` (Custom password), and `'passphrase'` (Passphrase/EFF diceware). The "passphrase" tab is Free-accessible. The "generate" tab generates a cryptographically secure password (not a diceware passphrase) — this is a password-class feature, so it should be Pro-locked.
   - What's unclear: CONTEXT.md uses "generated passphrases" loosely. The existing `'generate'` tab is a password generator (not diceware). The `'passphrase'` tab is the EFF diceware passphrase. The decision "Free: generated passphrase only" means the `'passphrase'` tab is Free.
   - Recommendation: Map `'passphrase'` tab → Free+ (locked for anonymous). Map `'generate'` and `'custom'` tabs → Pro only. This interpretation aligns with "Free users can only generate passphrases (no custom passwords)" from the phase goal.

2. **Should the protection_type field use snake_case or camelCase?**
   - What we know: CONTEXT.md specifies `protection_type` (snake_case). The existing schema fields use camelCase in TypeScript (`expiresIn`, `passwordHash`) but the API JSON uses whatever Zod serializes. The existing `CreateSecretSchema` uses `expiresIn` (camelCase), `password` (single word). The CONTEXT.md decision is explicit: `'none' | 'passphrase' | 'password'` as the enum, field name `protection_type`.
   - Recommendation: Use `protection_type` (snake_case) as specified in CONTEXT.md. This is an explicit user decision.

3. **Does the server need to look up `subscriptionTier` for passphrase requests?**
   - What we know: Passphrase is Free+. The server only needs to verify the user is authenticated (userId exists), not their tier.
   - Recommendation: For `protection_type === 'passphrase'`, only check `!!userId`. For `protection_type === 'password'`, check userId AND subscriptionTier === 'pro'. This avoids a DB lookup for passphrase-protected secrets from free users.
