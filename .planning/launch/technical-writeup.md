# How we built a secret sharer where we mathematically cannot read your data

We can't read your secrets. Not even under subpoena.

That's not a privacy policy. It's an architectural property. Here's exactly why it's true — four design decisions that together produce a system where the server is cryptographically blind to the plaintext it stores.

---

## Section 1: The URL Fragment Mechanism (RFC 3986 §3.5)

Everything rests on this.

The URI fragment — the `#...` part of a URL — is defined in RFC 3986 §3.5 as a client-side reference. The spec defines it as an identifier for a secondary resource within the primary document, resolved entirely by the client. The browser HTTP stack is required to strip the fragment before constructing the HTTP request. It never appears in the request-line, the Host header, or any request header. It is gone before the TCP connection is established.

This is not a feature we configured. It is how the URI specification defines fragment handling across every compliant browser. Chromium, Firefox, Safari, and every HTTP client that implements RFC 3986 exhibit this behavior.

The practical consequence for Torch Secret: the 256-bit encryption key lives in the URL fragment — `https://torchsecret.com/reveal/[id]#[43-char-key]`. The server's access log receives only `/reveal/[id]`. The key never appears in any server log, request header, CDN access log, or database access log — regardless of how the server is instrumented.

A common objection: "What stops the server from reading the URL before the fragment is stripped?" The answer is that there is no such moment. The fragment is stripped by the browser's HTTP implementation before the socket is opened. No server-side code, middleware, reverse proxy, or network tap can observe it at any point in the request lifecycle.

The result: the server receives an encrypted blob from the sender and later delivers that blob to the recipient. It never possesses the key to decrypt either payload.

## Section 2: AES-256-GCM + IV Construction

**Why GCM mode, specifically.**

CBC and CTR modes without a MAC allow a network attacker or the server itself to flip bits in the ciphertext. The recipient decrypts corrupted plaintext and has no way to know. GCM's 128-bit authentication tag makes tampering detectable: if a single bit of the ciphertext has changed since encryption, the `crypto.subtle.decrypt` call throws and decryption is aborted. The server cannot silently corrupt your secret.

**Key generation.**

```typescript
const key = await crypto.subtle.generateKey(
  { name: 'AES-GCM', length: 256 },
  true, // extractable: must export for URL embedding
  ['encrypt', 'decrypt'],
);
const rawKey = await crypto.subtle.exportKey('raw', key);
const keyBase64Url = uint8ArrayToBase64Url(new Uint8Array(rawKey));
// Result: 43-character base64url string (256 bits = 32 bytes)
```

The key is generated by the OS CSPRNG via the Web Crypto API. No `Math.random`, no userspace PRNG. The `true` extractable flag is necessary only to embed the key in the URL — on the recipient side, the key is imported as non-extractable and decrypt-only (the `CryptoKey` object cannot be re-exported after import).

**IV construction.**

A fresh 96-bit IV is generated per secret using `crypto.getRandomValues`. NIST recommends 96-bit for GCM's counter-based construction; longer IVs get hashed down to 96 bits internally, providing no additional security. Reusing an IV with the same key in GCM is catastrophic: the two ciphertexts XOR to reveal the auth key material, breaking authentication entirely. We generate a new key AND a new IV for every secret.

```typescript
const iv = crypto.getRandomValues(new Uint8Array(12)); // 96-bit IV

const ciphertextBuffer = await crypto.subtle.encrypt(
  { name: 'AES-GCM', iv },
  key,
  paddedBytes, // PADME-padded plaintext
);

// Prepend IV: [IV 12 bytes][ciphertext + 16-byte auth tag]
const combined = new Uint8Array(iv.length + ciphertextBuffer.byteLength);
combined.set(iv, 0);
combined.set(new Uint8Array(ciphertextBuffer), iv.length);

const ciphertext = uint8ArrayToBase64(combined); // standard base64, sent to server
```

**Blob structure:** `[IV 12 bytes][ciphertext][auth tag 16 bytes]`, encoded as standard base64. This is a different encoding from the key (base64url for URL embedding). The combined blob is stored as a text column in PostgreSQL.

On the recipient side, the key is imported as non-extractable and decrypt-only:

```typescript
return crypto.subtle.importKey(
  'raw', rawKey, 'AES-GCM',
  false,       // non-extractable: no re-export after import (defense in depth)
  ['decrypt'], // decrypt-only: recipient side restriction
);
```

After import, the browser cannot serialize or re-export the `CryptoKey` object. This is defense-in-depth: even if an attacker can execute arbitrary JavaScript in the tab, the key cannot be exfiltrated via the Web Crypto API.

## Section 3: PADME Padding

**Why ciphertext length is a real threat.**

An attacker observing ciphertext lengths at the network layer can bucket secrets: "this blob is definitely under 30 bytes — probably a short PIN." For a service designed to carry high-value credentials (API keys, passwords, service tokens), length leakage is a meaningful signal even when the content is fully encrypted.

**Why not power-of-2 rounding.**

The naive approach rounds to the next power of 2: a 33-byte plaintext rounds to 64 bytes (94% overhead), a 65-byte plaintext rounds to 128 bytes (97% overhead). Beyond the waste, it creates a perverse incentive: adding two bytes to a 63-byte secret doubles the padded size. For a service handling API keys and passwords — which cluster in the 20–100 byte range — power-of-2 rounding is both expensive and inconsistent.

**PADME (from the PURBs paper, PETS 2019).**

PADME pads plaintext to the nearest PADME boundary, guaranteeing at most 12% overhead for any input size. The algorithm:

```typescript
// Reference: PADME algorithm from PURBs paper (PETS 2019)
// https://lbarman.ch/blog/padme/
// At most 12% overhead (vs. 100% for power-of-2 rounding)
export function padmeLength(len: number): number {
  if (len <= MIN_PADDED_SIZE) return MIN_PADDED_SIZE;
  const e = Math.floor(Math.log2(len));
  const s = Math.floor(Math.log2(e)) + 1;
  const lastBits = e - s;
  const bitMask = (1 << lastBits) - 1;
  return (len + bitMask) & ~bitMask;
}
```

**Floor at 256 bytes (`MIN_PADDED_SIZE`).**

Every plaintext under 256 bytes is padded to exactly 256 bytes before encryption. For the most common secrets — API keys, passwords, tokens — this produces identical-length ciphertext regardless of actual length. Length leakage is eliminated entirely for the most common case. Secrets above 256 bytes get PADME-boundary rounding with at most 12% overhead.

## Section 4: Atomic Destroy Transaction

**The race condition without a transaction.**

Without a transaction, two concurrent GET requests both SELECT the row, both see the ciphertext, both receive the plaintext, and both fire DELETE — the second DELETE is a no-op. The secret is revealed twice, to two different recipients. This is not hypothetical: under any meaningful load, two legitimate clients could click the link within milliseconds of each other.

**The three-step transaction.**

```typescript
// retrieveAndDestroy() — three-step transaction prevents race condition
return db.transaction(async (tx) => {
  // Step 1: SELECT — read ciphertext
  const [secret] = await tx.select().from(secrets).where(eq(secrets.id, id));

  // Step 2: ZERO — overwrite ciphertext with '0' characters
  // (mitigates WAL data remanence)
  await tx.update(secrets)
    .set({ ciphertext: '0'.repeat(secret.ciphertext.length) })
    .where(eq(secrets.id, id));

  // Step 3: DELETE (anonymous) or UPDATE status='viewed' (user-owned)
  await tx.delete(secrets).where(eq(secrets.id, id));

  return secret; // returns original ciphertext from Step 1
});
```

**Why zero before delete (WAL remanence).**

PostgreSQL's write-ahead log stores the pre-image of every row before it is modified. When the DELETE commits, the WAL segment still contains the original ciphertext — until the segment is recycled or VACUUM runs. The zeroing step in Step 2 means that the WAL copy contains `'0000...'` rather than the original ciphertext. This is defense-in-depth against filesystem forensics on the database host: a full disk image of the database host after a DELETE would not expose the original plaintext.

**Password-protected secrets.**

For password-protected secrets, `verifyAndRetrieve()` adds a Step 0: `SELECT ... FOR UPDATE` — a pessimistic row-level lock. Drizzle ORM has no native `FOR UPDATE` support, so we use `tx.execute(sql\`SELECT id FROM secrets WHERE id = ${id} FOR UPDATE\`)`. The subsequent Drizzle SELECT in Step 1 inherits the lock within the same transaction. A second concurrent caller blocks on the lock until the first transaction commits. At that point, the row is gone — the second caller sees no row and receives a 404. This enforces the "one successful verification attempt" guarantee even under concurrent load.

## Section 5: Defense in Depth Beyond Crypto

The zero-knowledge invariant extends beyond the cryptographic operations. No log line, database record, or analytics event in the system may contain both a `userId` and a `secretId` in the same payload.

This matters because a combined record would allow an attacker with access to the database or logs to correlate which user created which secret — breaking the anonymity guarantee even if the ciphertext remains unbroken. The `secrets.user_id` column is a nullable FK to `users.id`, but `secrets.id` is never stored in the users or sessions tables. The Pino logger redacts secret IDs from URL paths via regex. PostHog's analytics client strips URL fragments via `sanitize_properties` before any event fires.

This invariant is enforced by code review convention and documented in `INVARIANTS.md` in the repository.

## Section 6: Honest Limitations

These are architectural properties of the system, not implementation bugs:

- **Device compromise** nullifies all guarantees — if RAM or the browser tab are accessible to an attacker, the plaintext and key are exposed. The zero-knowledge property protects against a compromised server, not a compromised client.
- **Malicious browser extensions** can intercept plaintext before the Web Crypto API encrypts it. There is no defense against extensions with access to page content.
- **Ciphertext length** is observable at the network layer (mitigated by PADME but not eliminated for very short secrets above the 256-byte floor). An attacker observing blob sizes gets PADME-rounded length, not exact length — but perfect length hiding requires fixed-length padding.
- **JS trust model:** users must trust the browser to run the correct client-side crypto code. The source is auditable at the GitHub link; there is no binary distribution to verify against.
