  <name>Task 1: Integration tests for API-level expiration enforcement</name>
  <files>server/src/routes/__tests__/expiration.test.ts</files>
  <action>
    Create `server/src/routes/__tests__/expiration.test.ts` following the exact same test setup pattern as the existing `secrets.test.ts`: import buildApp, db, pool, secrets, sql; create fresh app in beforeEach; ensure table in beforeAll; clean up in afterEach; close pool in afterAll.

    **Strategy for testing expired secrets:** Insert secrets directly into the database with an `expiresAt` timestamp in the PAST (e.g., `new Date(Date.now() - 60_000)` for 1 minute ago). This bypasses the API's `expiresIn` enum validation and lets us test expiration behavior immediately without waiting.

    Use `db.insert(secrets).values(...)` to directly insert rows with past expiresAt.

    Tests to write:

    **1. "GET /api/secrets/:id returns 404 for expired non-password secret"**
    - Insert secret with past expiresAt directly via db.insert
    - GET /api/secrets/:id -> expect 404
    - Verify response body matches SECRET_NOT_AVAILABLE shape

    **2. "GET /api/secrets/:id/meta returns 404 for expired secret"**
    - Insert secret with past expiresAt
    - GET /api/secrets/:id/meta -> expect 404
    - Verify response body matches SECRET_NOT_AVAILABLE shape

    **3. "POST /api/secrets/:id/verify returns 404 for expired password-protected secret"**
    - Insert secret with past expiresAt AND a passwordHash (use a known argon2id hash or import hashPassword from password.service.ts to create one)
    - POST /api/secrets/:id/verify with any password -> expect 404
    - Verify response body matches SECRET_NOT_AVAILABLE shape

    **4. "expired secret 404 response is identical to nonexistent secret response (anti-enumeration)"**
    - Insert expired secret
    - GET the expired secret -> 404, capture body
    - GET a nonexistent 21-char ID -> 404, capture body
    - Assert both bodies are deepEqual
    - Repeat for /meta endpoint
    - Repeat for /verify endpoint (with password body)

    **5. "non-expired secret is still retrievable (expiration guard does not block valid secrets)"**
    - Create a secret via POST /api/secrets (normal flow, future expiration)
    - GET /api/secrets/:id -> expect 200 with ciphertext
    - This confirms the expiration guard doesn't break normal retrieval

    **6. "expired secret is cleaned up from database after retrieval attempt"**
    - Insert secret with past expiresAt
    - GET /api/secrets/:id -> 404
    - Direct DB query: verify row no longer exists (the inline cleanup in the service should have deleted it)

    **7. "expired password-protected secret is cleaned up from database after verify attempt"**
    - Insert expired password-protected secret
    - POST /api/secrets/:id/verify -> 404
    - Direct DB query: verify row no longer exists

    For tests needing a passwordHash: import `hashPassword` from `../../services/password.service.js` and hash a test password in a beforeAll or per-test.

    Use nanoid() or a hardcoded 21-char string for direct-insert IDs. If using nanoid, import from 'nanoid'.

    - Non-expired secrets are unaffected by expiration guards

<task type="auto">
  <name>Task 2: Integration tests for the expiration worker bulk cleanup</name>
  <files>server/src/workers/__tests__/expiration-worker.test.ts</files>
  <action>
    Create `server/src/workers/__tests__/expiration-worker.test.ts`. This tests the worker's cleanup logic directly -- NOT via the cron scheduler (which would require waiting 5 minutes or mocking timers).

    **Approach:** The worker module's cron callback contains the bulk cleanup logic. To test it directly, either:
    - (a) Extract the cleanup logic into a separate exported function (e.g., `export async function cleanExpiredSecrets(): Promise<number>`) that the cron callback calls. The test imports and calls this function directly. This is the preferred approach -- it keeps the cron scheduling separate from the testable business logic.
    - (b) If the 06-01 plan did not extract the function, modify the worker module to export the cleanup function.

    **Important:** Update `server/src/workers/expiration-worker.ts` if needed to export a `cleanExpiredSecrets()` function that performs the two-step UPDATE+DELETE and returns the count of deleted rows. The `startExpirationWorker` function would then call `cleanExpiredSecrets()` inside the cron callback.

    Test setup: Same pattern as secrets.test.ts (db, pool, secrets, sql imports; table creation; cleanup; pool close). Do NOT start the cron scheduler in tests.

    Tests to write:

    **1. "cleanExpiredSecrets deletes expired secrets"**
    - Insert 2 secrets with past expiresAt via db.insert
    - Call cleanExpiredSecrets()
    - Direct DB query: both rows are gone
    - Return value should indicate 2 deletions (if rowCount works; otherwise verify via DB count)

    **2. "cleanExpiredSecrets leaves non-expired secrets intact"**
    - Insert 1 expired secret and 1 non-expired secret (expiresAt in the future)
    - Call cleanExpiredSecrets()
    - Direct DB query: expired secret gone, non-expired secret still exists with original ciphertext

    **3. "cleanExpiredSecrets zeros ciphertext before deletion"**
    - This is harder to test directly since zero+delete happens in sequence. Two approaches:
      - (a) Inspect the WAL (impractical)
      - (b) Add a test that verifies the two-step pattern by checking that between UPDATE and DELETE, the ciphertext is zeroed. This requires splitting the function or observing intermediate state.
      - (c) Simplest: verify that after cleanup, no row exists AND (if the UPDATE+DELETE are in separate statements, not a transaction) temporarily break the DELETE to confirm the UPDATE happens first. This is fragile.
      - **Recommended:** Test that the function runs without error and deletes the rows. The ciphertext zeroing is a defense-in-depth measure verified by code review, not runtime testing. Document in test comment: "Ciphertext zeroing verified by code inspection -- bulk UPDATE precedes DELETE in expiration-worker.ts".

    **4. "cleanExpiredSecrets handles empty table gracefully"**
    - Ensure table is empty
    - Call cleanExpiredSecrets()
    - No error thrown, returns 0 (or equivalent)

    **5. "cleanExpiredSecrets handles mix of password-protected and non-password expired secrets"**
    - Insert 1 expired non-password secret and 1 expired password-protected secret
    - Call cleanExpiredSecrets()
    - Both are deleted (the worker doesn't distinguish -- it cleans all expired secrets regardless of password status)

    After writing tests, run the full test suite to confirm no regressions: `npm run test:run`
