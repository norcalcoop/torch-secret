  <files>server/src/tests/auth.test.ts</files>
  <action>
    Check if a tests directory exists at server/src/tests/ — if not, create it.
    Check existing test patterns in the codebase (look for existing *.test.ts in server/src/).
    Follow the same test setup pattern (buildApp, beforeEach cleanup, etc.).

    Create server/src/tests/auth.test.ts with the following test suites:

    Test setup:
    - Import buildApp from ../app.js
    - Import db from ../db/connection.js and schema tables for cleanup
    - Import supertest
    - Create app instance once via buildApp()
    - In beforeEach: clean up test users from DB (DELETE FROM users WHERE email LIKE '%@test-auth.example.com')
      Use a distinct email domain (@test-auth.example.com) to avoid colliding with other test data.
    - In afterAll: close any open resources if needed.

    IMPORTANT: NODE_ENV=test disables requireEmailVerification (set in auth.ts).
    This means test users can sign in immediately after sign-up without email verification.
    Tests should assert this behavior.

    Helper function: signUpAndGetCookie(email, password, name) → Promise<string>
      POST /api/auth/sign-up/email → extract better-auth.session_token= cookie from set-cookie header.
      (Pattern from research: GitHub discussion #2299)

    Test Suite 1: Sign up (AUTH-01)
    - "POST /api/auth/sign-up/email with valid credentials returns 200"
      body: { email: 'signup@test-auth.example.com', password: 'testpass123', name: 'Test User' }
      expect: status 200
      expect: response headers['set-cookie'] to contain a string starting with 'better-auth.session_token='
    - "POST /api/auth/sign-up/email with duplicate email returns error status (422)"
      Sign up once, sign up again with same email.
      expect: second call returns 422 status (or 400 — check actual response and assert accordingly)

    Test Suite 2: Sign in (AUTH-03)
    - "POST /api/auth/sign-in/email with valid credentials returns 200 and session cookie"
      First sign up the user, then sign in.
      expect: status 200
      expect: set-cookie contains better-auth.session_token
    - "POST /api/auth/sign-in/email with wrong password returns 401 or 422"
      Sign up, then sign in with wrong password.
      expect: status 401 or 422 (assert non-200)

    Test Suite 3: Session (AUTH-05)
    - "GET /api/me with valid session cookie returns 200 with user data"
      Use signUpAndGetCookie helper.
      Make GET /api/me with Cookie header set to the session cookie.
      expect: status 200
      expect: body.user.email === test email
      expect: body.user.name === 'Test User'
      expect: body.user.id is a non-empty string
    - "GET /api/me without session cookie returns 401"
      GET /api/me with no Cookie header.
      expect: status 401

    Test Suite 4: Sign out (AUTH-08)
    - "POST /api/auth/sign-out with valid session destroys session"
      Sign up → get cookie → sign out (POST /api/auth/sign-out with Cookie header) →
      GET /api/me with same cookie → expect 401.
      (Session cookie should be invalid after sign-out)

    Test Suite 5: Password reset request (AUTH-04)
    - "POST /api/auth/forget-password with valid email returns 200"
      Note: Better Auth's password reset endpoint may be at /api/auth/forget-password or
      /api/auth/reset-password. Check by inspecting auth.api routes or use the documented
      client method's underlying endpoint. From research: the endpoint the client calls is
      /api/auth/forget-password (the REQUEST endpoint, not the completion endpoint).
      body: { email: existingEmail, redirectTo: '/reset-password' }
      expect: status 200 (regardless of whether email exists — prevents enumeration)
    - "POST /api/auth/forget-password with non-existent email returns 200 (no enumeration)"
      body: { email: 'nonexistent@test-auth.example.com', redirectTo: '/reset-password' }
      expect: status 200 (same response either way)

    Test Suite 6: Zero-knowledge invariant (structural)
    - "GET /api/me response body does not contain secretId field"
      Sign up → get session → GET /api/me → assert body.user does not have 'secretId' property.
      This is a structural test ensuring the invariant is not accidentally violated.

    OAuth tests (AUTH-06, AUTH-07):
    - OAuth flows are browser-redirect-based and cannot be integration-tested with Supertest alone.
    - Add a placeholder test with a comment:
      it.skip('OAuth flows tested manually via browser checkpoint', ...)
    - The human-verify checkpoint (Task 2) covers these manually.

    Test file location: server/src/tests/auth.test.ts
    Vitest config for server tests uses node environment. Ensure the test file is in a location
    that the existing vitest config picks up (check vitest.config.ts for the server project
    include patterns).
