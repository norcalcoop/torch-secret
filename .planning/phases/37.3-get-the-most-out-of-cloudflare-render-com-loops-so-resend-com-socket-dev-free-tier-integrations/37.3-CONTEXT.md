# Phase 37.3: Free Tier Integrations - Context

**Gathered:** 2026-02-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Configure and fully utilize the free tiers of five services already partially in use: Cloudflare, Render.com, Loops.so, Resend.com, and Socket.dev. No new application features. This phase is about infrastructure completeness — switching on capabilities that are available but unused, closing configuration gaps, and ensuring each service is properly wired end-to-end.

Each service has a different maturity level going in:
- **Resend.com** — Most complete: Audience sync is wired (confirm → add, unsubscribe → mark). Only gap is render.yaml parity.
- **Loops.so** — One gap: confirmed subscribers (email capture flow) are not synced to Loops.
- **Render.com** — Configuration gap: render.yaml is missing ~8 env var stubs vs .env.example.
- **Cloudflare** — Proxied (orange cloud) but no page rules, caching config, or security settings in the repo.
- **Socket.dev** — Zero integration. GitHub app not installed, no documentation.

</domain>

<decisions>
## Implementation Decisions

### Cloudflare — caching strategy
- Cache `/assets/*` for 1 year (Vite-hashed filenames make this safe — content changes = new filename)
- HTML responses (`/`, `/create`, `/reveal/*`, etc.) cached short or bypassed — they are the SPA entry point and must stay fresh
- Configure as Cloudflare Cache Rules (not legacy Page Rules)

### Cloudflare — security posture
- Enable **Bot Fight Mode** (free tier) — blocks known bot traffic at the edge
- Enable **DDoS protection** (on by default for proxied domains, just verify it's active)
- Do NOT add Cloudflare rate limiting rules — the app has Express + Redis rate limiting already; two rate limiters fighting causes more problems than it solves
- No WAF custom rules needed at this stage

### Cloudflare — analytics
- Enable Cloudflare Web Analytics (free, privacy-friendly, no JS snippet needed for proxied domains)
- Provides traffic visibility without relying on client-side PostHog for infrastructure metrics

### Cloudflare Worker — uptime ping (dual purpose)
- Create a **Cloudflare Worker with a cron trigger** that pings `https://[app-domain]/api/health` every 10 minutes
- Dual purpose: (1) prevents Render.com free tier spin-down (~60s cold starts), (2) demonstrates Cloudflare Workers free tier usage
- Worker lives in the repo (`workers/keep-alive/`) with a `wrangler.toml` config
- Ping is fire-and-forget — no alerting, no state. Just a GET request that keeps the service warm.

### Render.com — env var parity
- render.yaml updated to full parity with `.env.example`
- All missing env vars added as `sync: false` stubs: `LOOPS_API_KEY`, `APP_URL`, `BETTER_AUTH_TRUSTED_ORIGINS`, `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `STRIPE_PRO_PRICE_ID`, `RESEND_AUDIENCE_ID`, `IP_HASH_SALT`, `POSTHOG_API_KEY` (if needed separately), plus any others in .env.example
- Values stay out of the repo — stubs are reminders for the Render dashboard

### Render.com — expiration worker
- Keep the expiration worker in-process (node-cron in the main server process) — no change
- No separate Render cron service; free tier limits make in-process the right choice

### Loops.so — subscriber sync (new)
- When a marketing subscriber confirms their email (double opt-in), fire:
  ```
  loops.sendEvent({ email, eventName: 'subscribed', contactProperties: { source: 'email-capture' } })
  ```
- This is separate from the `registered` event (which fires on full account creation)
- Zero-knowledge safe: only email + source property, no userId or secretId
- Call goes in `subscribers.service.ts` `confirmSubscriber()` — alongside the existing Resend Audience sync

### Loops.so — account deletion (new)
- When a user deletes their account, call `loops.deleteContact({ email })` before removing the DB record
- GDPR hygiene: prevents sending emails to deleted accounts
- Wire into the account deletion flow (wherever Better Auth user deletion is handled)

### Loops.so — what NOT to add
- No secret lifecycle events (created, viewed, expired) — too frequent, and risks zero-knowledge invariant
- No separate unsubscribe event to Loops — Loops manages unsubscribes via its own links; the `deleteContact` on account deletion covers the GDPR case

### Resend.com — status
- Audience sync is already fully wired in `subscribers.service.ts`
- No new Resend code changes needed
- render.yaml parity (above) covers the missing `RESEND_AUDIENCE_ID` stub

### Socket.dev — integration approach
- Install the **Socket.dev GitHub App** via the GitHub repo settings (or github.com/apps/socket-security)
- No code changes required — Socket reads `package.json` and `package-lock.json` automatically
- GitHub app posts PR comments and checks when a dependency change introduces supply chain risk

### Socket.dev — config file
- Add a minimal `socket.yml` to the repo root to document the policy intent (even if defaults are fine)
- Marking known-safe exceptions or adjusting thresholds signals intentionality to future contributors
- Claude's Discretion: determine if any existing dependencies trigger Socket warnings and whether they need explicit allowlisting

### Socket.dev — documentation
- Add a brief note in the README security section: "Dependency changes are automatically scanned by Socket.dev on PRs for supply chain security."

### CI auto-triggers — re-enable
- Re-enable GitHub Actions CI auto-triggers: `on: push` to `main` and `on: pull_request`
- Currently `workflow_dispatch`-only, which means Socket.dev PR checks are the only automated gate on dependency changes — CI should also run
- Claude's Discretion: decide if branch filtering (e.g., paths-ignore for docs) is appropriate

### Claude's Discretion
- Whether `socket.yml` needs any package allowlisting (requires inspecting current `package-lock.json` against Socket's database)
- Exact Cloudflare Cache Rule TTLs for HTML (5 min, 1 hour, or bypass — depends on deploy frequency)
- Whether to use `wrangler deploy` in CI or document as a manual deploy step for the Cloudflare Worker
- Email template management in Resend vs inline HTML (current inline approach is fine unless team wants visual template editing)

</decisions>
