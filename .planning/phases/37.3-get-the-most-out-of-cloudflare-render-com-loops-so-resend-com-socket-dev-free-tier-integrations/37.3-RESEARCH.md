## Summary

This phase has five distinct work streams: (1) Cloudflare configuration, (2) a Cloudflare Worker keep-alive cron job, (3) Render.com render.yaml parity, (4) Loops.so subscriber sync + account deletion, and (5) Socket.dev GitHub App + socket.yml. Most streams are configuration-only; the two code-heavy items are the Cloudflare Worker (new TypeScript file) and account deletion (new backend route + frontend modal).

The account deletion flow is the most technically complex item. Better Auth 1.x has a built-in `/delete-user` endpoint but it requires opt-in via `user.deleteUser.enabled: true` in the `betterAuth()` config. The recommended approach for this codebase is to use the `beforeDelete` hook to fire Loops cleanup and null out secrets, then let Better Auth handle the actual user row deletion. This avoids re-implementing the session/cookie cleanup logic that Better Auth's `internalAdapter.deleteUser()` + `deleteSessions()` + `deleteSessionCookie()` handles internally.

The CI re-enablement is a one-line YAML change but has a cascading effect: it activates Render's `autoDeployTrigger: checksPass` gate, meaning pushes to main will trigger a production deploy only after CI passes. This makes it the most deployment-critical change in the phase.

**Primary recommendation:** Implement in this order: (1) render.yaml parity and CI re-enable first (no risk, enables production deploy gate), (2) Loops subscriber sync (tiny addition to existing function), (3) socket.yml + GitHub App (config only), (4) Cloudflare cache rules + bot fight mode (dashboard config only), (5) Cloudflare Worker (new codebase artifact), (6) account deletion (most complex — backend + frontend).

### Pattern 1: Cloudflare Worker Cron Trigger (TypeScript)

**What:** A scheduled Worker that fires every 10 minutes and GETs `/api/health` to prevent Render cold starts.
**When to use:** This exact pattern — keep-alive with no state, no env bindings needed.
**Example:**
```typescript
// workers/keep-alive/src/index.ts
// Source: https://developers.cloudflare.com/workers/examples/cron-trigger/

const TARGET_URL = 'https://torchsecret.com/api/health';

export default {
  async scheduled(
    _controller: ScheduledController,
    _env: Record<string, never>,
    ctx: ExecutionContext,
  ): Promise<void> {
    ctx.waitUntil(
      fetch(TARGET_URL).catch(() => {
        // Fire-and-forget: ignore errors — Worker is best-effort ping only
      }),
    );
  },
};
```

```toml
# workers/keep-alive/wrangler.toml
name = "torch-secret-keep-alive"
main = "src/index.ts"
compatibility_date = "2026-02-28"
# account_id: set via CLOUDFLARE_ACCOUNT_ID env var or wrangler login

[triggers]
crons = ["*/10 * * * *"]
```

**Cron free tier limit:** 5 cron triggers per account (this worker uses 1).
**CPU limit on free tier:** 10ms per invocation — a single `fetch()` call is well within this.
**Testing locally:** `npx wrangler dev --test-scheduled` exposes `/__scheduled` endpoint.
**Deploy command:** `wrangler deploy` (from `workers/keep-alive/` directory, or with `--config workers/keep-alive/wrangler.toml`).

### Pattern 2: Better Auth `deleteUser` with `beforeDelete` Hook

**What:** Enable Better Auth's built-in `/delete-user` endpoint and use `beforeDelete` to fire Loops cleanup + secrets null-out before the user row is deleted.
**When to use:** User account deletion with GDPR cleanup — the standard Better Auth pattern.

**Critical finding:** The `deleteUser` endpoint is **disabled by default**. It must be explicitly enabled:

```typescript
// server/src/auth.ts — extend the betterAuth() call
user: {
  additionalFields: { /* ... existing ... */ },
  deleteUser: {
    enabled: true,
    beforeDelete: async (user) => {
      // 1. GDPR: delete Loops contact (fire-and-forget — deletion must not fail because of Loops)
      await loops.deleteContact({ email: user.email }).catch((err: unknown) => {
        logger.error(
          { err: err instanceof Error ? err.message : String(err) },
          'Loops deleteContact failed on account deletion',
        );
      });

      // 2. ZK invariant: null out secrets.user_id so shared links keep working
      await db
        .update(secrets)
        .set({ userId: null })
        .where(eq(secrets.userId, user.id));
    },
  },
},
```

**What Better Auth does internally after `beforeDelete`:**
1. Calls `internalAdapter.deleteUser(user.id)` — deletes the user row (cascade deletes sessions, accounts via ON DELETE CASCADE)
2. Calls `internalAdapter.deleteSessions(user.id)`
3. Calls `deleteSessionCookie(ctx)` — clears the auth cookie
4. Returns `{ success: true, message: "User deleted" }`

**Important session freshness gate:** The built-in endpoint checks session freshness by default (`freshAge !== 0`). If the session is older than the freshAge threshold (default: 24 hours), it throws `SESSION_EXPIRED`. For this UX pattern (user clicks "Delete Account" → types `delete` → confirms), the user is presumably in an active session. This freshness check is a security feature — do not disable it.

**How to trigger deletion from the Express route (`DELETE /api/me`):**
Since the frontend confirmation modal calls our own `DELETE /api/me` endpoint (not directly POST to `/api/auth/delete-user`), the `DELETE /api/me` handler must forward to Better Auth's endpoint. Two options:

Option A — Use `auth.api.deleteUser()` with `Request` forwarding:
```typescript
// server/src/routes/me.ts — new DELETE handler
meRouter.delete('/', requireAuth, async (req, res) => {
  const user = res.locals.user as AuthUser;
  // Convert Express req to a standard Request for Better Auth
  // ... then call auth.api.deleteUser(...)
});
```

Option B — Skip the Better Auth endpoint entirely and handle deletion directly:
```typescript
// Delete via Drizzle directly (users table has ON DELETE CASCADE for sessions/accounts)
// After: null secrets.user_id, call loops.deleteContact, then db.delete(users)
// Still need to clear the session cookie manually
```

**Recommendation: Option A (auth.api.deleteUser) is cleaner** — Better Auth handles cookie clearing. However, the `auth.api.deleteUser` server-side call requires passing headers for session extraction. The `fromNodeHeaders` + `auth.api.deleteUser` pattern is the same as `auth.api.getSession` already used in `require-auth.ts`.

The simplest approach: the `DELETE /api/me` route calls `auth.api.deleteUser` with the original request headers. Better Auth runs `beforeDelete` (which does the Loops + secrets cleanup), then deletes the user and clears the cookie. The route handler returns `{ ok: true }` to the frontend, which then redirects to `/`.

### Pattern 3: Loops `deleteContact` API

**What:** Remove a contact from Loops on account deletion (GDPR hygiene).
**Signature (from installed `loops@6.2.0` type definitions):**

**Pattern from existing codebase (fire-and-forget in `beforeDelete`):**
```typescript
await loops.deleteContact({ email: user.email }).catch((err: unknown) => {
  logger.error({ err: err instanceof Error ? err.message : String(err) },
    'Loops deleteContact failed on account deletion');

```
Note: `beforeDelete` is `async`, so `await` on the Loops call is appropriate here. It is not fire-and-forget (unlike the Loops calls in `databaseHooks` which must be non-async to avoid `@typescript-eslint/require-await`). Using `await + .catch()` means Loops failure is logged but does not block deletion.

### Anti-Patterns to Avoid

- **Do NOT call `db.delete(users).where(...)` directly for account deletion.** Better Auth manages session cookie clearing, session table cleanup, and account row deletion internally. Bypassing it leaves stale cookies that still authenticate.
- **Do NOT pass `userId` to `loops.deleteContact`.** Only pass `email`. The `userId` field in the Loops API refers to an external system user ID — not our Better Auth userId. Passing our `user.id` there creates a mapping in Loops that we don't want.
- **Do NOT put the Worker's target URL in `wrangler.toml` env vars.** The CONTEXT.md decision is explicit: hardcode in the script. The Worker has one job.
- **Do NOT use `wrangler.toml` `[vars]` for secrets.** Only `[secrets]` via `wrangler secret put` is safe for sensitive values (not needed here — no secrets in this Worker).
- **Do NOT use `sync: false` inside a Render environment group.** Render ignores `sync: false` inside `envVarGroups`. The existing render.yaml uses inline `envVars` on the web service — that pattern is correct and must be preserved.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Session cookie clearing on user delete | Manual `res.clearCookie()` calls | Better Auth `auth.api.deleteUser` with `beforeDelete` | Better Auth knows its own cookie names, signing keys, and domain settings |
| Session DB cleanup on delete | `db.delete(sessions).where(userId)` | Better Auth `internalAdapter.deleteSessions()` via the deleteUser flow | The cascade FK (`onDelete: 'cascade'`) on sessions also handles this, but Better Auth's cleanup is belt-and-suspenders |
| Cron scheduling for keep-alive | Express endpoint + external cron | Cloudflare Worker scheduled trigger | Workers cron is free, serverless, needs no DB/state, and survives app restarts |
| Cache invalidation logic | Custom purge scripts | Vite content-hashed filenames + 1-year Cache Rule | Hash in filename IS the cache buster — no invalidation needed |

**Key insight:** Better Auth's `deleteUser` flow (with `beforeDelete`) is a complete, tested deletion pipeline. The only custom logic this codebase needs is (1) null out `secrets.user_id` and (2) call `loops.deleteContact` — both belong in `beforeDelete`. Everything else is Better Auth's responsibility.

## Common Pitfalls

### Pitfall 1: Better Auth `deleteUser` Not Enabled by Default

**What goes wrong:** Calling `DELETE /api/auth/delete-user` or `auth.api.deleteUser()` returns a 404 error silently.
**Why it happens:** The endpoint exists in the built-in API but is gated behind `user.deleteUser.enabled: true`. From the source: `if (!ctx.context.options.user?.deleteUser?.enabled) { throw new APIError("NOT_FOUND"); }`.
**How to avoid:** Add `deleteUser: { enabled: true, beforeDelete: ... }` to the `user:` block in `auth.ts`.
**Warning signs:** 404 response to POST `/api/auth/delete-user`.

### Pitfall 2: `beforeDelete` Logging the User's Email

**What goes wrong:** Logging `user.email` in the `beforeDelete` error handler creates a log line with PII adjacent to the deletion event.
**Why it happens:** The `beforeDelete` callback receives the full user object.
**How to avoid:** Follow the existing pattern from `databaseHooks.user.create.after` — log only `err.message`, never `user.email`, `user.id`, or any combination that links identity to the deletion event.
**Reference:** `auth.ts` line 145-150 — the Loops enrollment error handler logs only `err.message`.

### Pitfall 3: `secrets.user_id` Null-Out Race Condition

**What goes wrong:** A user creates a secret between the `beforeDelete` null-out and the actual user deletion, leaving a dangling `user_id` reference.
**Why it happens:** `beforeDelete` runs before `internalAdapter.deleteUser()` — there's a small window.
**How to avoid:** The FK on `secrets.user_id` is `onDelete: 'set null'`. This means PostgreSQL automatically nulls out remaining `user_id` references when the user row is deleted (even if a secret was created in the window). The explicit null-out in `beforeDelete` is defense-in-depth for the existing rows. **Both mechanisms together** (explicit null-out + FK cascade) provide complete coverage.
**Confirmation from schema.ts:** `userId: text('user_id').references(() => users.id, { onDelete: 'set null' })`.

### Pitfall 4: Cloudflare Cache Rules — Free Tier TTL Limits

**What goes wrong:** Setting Edge TTL or Browser TTL to 1 year (31,536,000 seconds) fails or gets silently capped.
**Why it happens:** The official docs say "values available depend on your plan" but do not specify the free tier maximum. One search result mentioned a "2-hour minimum TTL" (possibly referring to something else). The 1-year duration (365 days) for assets is a Cloudflare Cache Rule behavior — since Vite hashes filenames, the cache does not need invalidation.
**How to avoid:** Configure the Cache Rule for `/assets/*` and check what TTL values the dashboard allows. If 1 year is unavailable, the maximum available value (e.g., 1 month, 1 year) is still safe because the filename hash guarantees content freshness.
**Confidence:** LOW — free tier TTL limit not confirmed in official docs. Verify in dashboard during implementation.

### Pitfall 5: Wrangler `account_id` in `wrangler.toml` is Optional

**What goes wrong:** Developer checks in `account_id` to `wrangler.toml`, exposing the Cloudflare account ID in the public repo.
**Why it happens:** Wrangler docs show it as a config option.
**How to avoid:** Use the `CLOUDFLARE_ACCOUNT_ID` environment variable (set in shell profile or CI) instead of hardcoding in `wrangler.toml`. The `wrangler.toml` can omit `account_id` entirely when the env var is set.
**Note:** The account ID is not a secret (it's visible in the Cloudflare dashboard URL), but omitting it from a checked-in config is cleaner.

### Pitfall 6: Render `sync: false` Only Prompts on Initial Blueprint Sync

**What goes wrong:** Developer adds a new `sync: false` stub and expects to be prompted in the Render dashboard on next Blueprint sync.
**Why it happens:** Per Render docs: "Render prompts you for these values only during the initial Blueprint creation flow. When you update an existing Blueprint, Render ignores any environment variables with `sync: false`."
**How to avoid:** After adding new `sync: false` stubs to render.yaml and pushing, manually navigate to Environment in the Render dashboard for the `torch-secret` service and add the new values there. The stubs in render.yaml serve as documentation only for existing deployments.

### Pitfall 7: Socket.yml `issueRules` is Deprecated

**What goes wrong:** Developer uses `issueRules: { someAlert: false }` to suppress a Socket warning, and gets a deprecation warning.
**Why it happens:** socket.yml v2 deprecated `issueRules` in favor of repository labels.
**How to avoid:** Use `version: 2` format. If a package needs allowlisting, check Socket's current v2 guidance for the replacement mechanism. The minimal `socket.yml` with `githubApp:` settings is sufficient to start.

### Pitfall 8: Worker `ctx.waitUntil()` vs Direct `await`

**What goes wrong:** Using `await fetch(...)` directly in the `scheduled()` handler without `ctx.waitUntil()` can cause the Worker to terminate before the fetch completes on some edge cases.
**Why it happens:** Cloudflare Workers' execution model terminates the isolate when the handler returns. `ctx.waitUntil()` extends the lifetime until the promise resolves.
**How to avoid:** Wrap the `fetch()` in `ctx.waitUntil(...)`. Since this is fire-and-forget, also add `.catch(() => {})` so a failed fetch does not cause the `waitUntil` promise to reject.

## Open Questions

1. **Cloudflare Cache Rule free tier TTL maximum**
   - What we know: Free tier supports Cache Rules (up to 10 rules). Browser TTL and Edge TTL are configurable. One source mentioned "2-hour minimum" but this conflicts with common practice.
   - What's unclear: Whether the free tier caps Edge TTL at some value less than 1 year (31,536,000 seconds) for `/assets/*`.
   - Recommendation: Attempt to set 1 year in the dashboard; if the UI rejects it, use the maximum available value. Document the actual TTL used in a comment on the Cache Rule.

2. **Socket.dev package warnings in this project**
   - What we know: Socket reads `package.json` and `package-lock.json` automatically after GitHub App install. Some packages may trigger supply-chain alerts.
   - What's unclear: Whether any current dependencies (e.g., Argon2id native bindings, older transitive dependencies) will trigger Socket warnings that require `issueRules` overrides.
   - Recommendation: Install the GitHub App first and review the initial scan results. If warnings appear for known-safe packages, add `issueRules` entries to `socket.yml`. If no warnings, the minimal config is sufficient.

3. **`auth.api.deleteUser()` server-side call signature**
   - What we know: Better Auth exposes `auth.api.deleteUser` (confirmed in type definitions). The pattern `auth.api.getSession({ headers: fromNodeHeaders(req.headers) })` is already established in `require-auth.ts`.
   - What's unclear: Whether `auth.api.deleteUser` accepts the same `{ headers }` parameter shape as `auth.api.getSession`, or requires a different format.
   - Recommendation: Check `node_modules/better-auth/dist/api/index.d.mts` for the exact `deleteUser` parameter type during implementation. If the `{ headers }` pattern does not apply (deleteUser might take `{ headers, body }` since it's a POST), use `fromNodeHeaders(req.headers)` for headers and pass an empty body. The `body.password` and `body.token` fields are optional.
