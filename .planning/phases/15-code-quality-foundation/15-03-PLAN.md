   This is expected to change many files since the codebase has never had Prettier. The changes are formatting-only (whitespace, quotes, trailing commas, Tailwind class ordering).

2. Run ESLint auto-fix to resolve auto-fixable lint issues:
   ```bash

   ```

3. Check for remaining ESLint errors that auto-fix cannot handle:
   ```bash
   npx eslint .
   ```

4. If there are remaining errors, fix them manually. Common type-aware rule violations to expect:
   - `@typescript-eslint/no-floating-promises` — add `void` prefix or `await` to unhandled promises
   - `@typescript-eslint/no-unsafe-assignment` — add type annotations to `any`-typed values
   - `@typescript-eslint/no-unsafe-member-access` — add type narrowing or assertions
   - `@typescript-eslint/no-unsafe-argument` — add type assertions at call sites
   - `@typescript-eslint/no-unsafe-call` — add type annotations to function references
   - `@typescript-eslint/require-await` — remove `async` from functions that don't await, or add `await`
   - `@typescript-eslint/no-misused-promises` — ensure Promise-returning functions aren't passed where void is expected (e.g., event handlers)
   - `@typescript-eslint/restrict-template-expressions` — add `.toString()` or type narrow before template literal interpolation

   For each remaining error, choose the MINIMAL fix:
   - Prefer adding type annotations over `// eslint-disable` comments
   - Use `// eslint-disable-next-line` ONLY if the rule is genuinely wrong for that line (e.g., test files with intentional `any` usage)
   - If a rule produces many false positives across the codebase (10+), consider adding a targeted rule override in eslint.config.ts for specific file patterns (e.g., test files may need relaxed rules)

5. Verify final state:
   ```bash
   npx eslint .
   npx prettier --check .
   npx tsc --noEmit
   ```
   All three must exit with code 0.

6. Run full test suite to confirm no behavioral regressions from formatting or lint fixes:
   ```bash
   npx vitest run
   ```
   All tests must pass. If any test fails, investigate whether the formatting/lint fix changed behavior (it shouldn't) or if the test is flaky.

Note: Formatting changes will touch many files. This is expected and intentional — it establishes the clean baseline. All future commits will maintain this formatting via the pre-commit hook.
