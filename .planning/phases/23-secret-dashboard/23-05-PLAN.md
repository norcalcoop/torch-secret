  <behavior>
Dashboard route tests (RED → GREEN → REFACTOR):

**Authentication guard tests:**
- GET /api/dashboard/secrets without session cookie → 401
- DELETE /api/dashboard/secrets/:id without session cookie → 401

**GET /api/dashboard/secrets (authenticated):**
- Returns 200 with `{ secrets: [] }` when user has no secrets
- Returns 200 with secrets belonging to the authenticated user (sorted newest first)
- Does NOT return secrets belonging to other users
- Response body for each secret NEVER contains `ciphertext` or `passwordHash` keys (DASH-05)
- Returns secrets of all statuses: active, viewed, expired, deleted

**DELETE /api/dashboard/secrets/:id (authenticated):**
- Returns 200 `{ success: true }` for an Active secret owned by the authenticated user
- After deletion: row still exists in DB with status='deleted' (soft-delete, not hard-delete)
- Returns 404 for a secret with status 'viewed' (cannot re-delete consumed secret)
- Returns 404 for a secret with status 'expired'
- Returns 404 for a secret owned by a different user (prevents cross-user deletion)
- Returns 404 for a non-existent secret ID

**Expiration worker soft-expire tests (new tests to add to existing file):**
- User-owned expired secret: after cleanExpiredSecrets(), row still exists with status='expired'
- User-owned expired secret: ciphertext is zeroed ('0')
- Anonymous expired secret: after cleanExpiredSecrets(), row is deleted (unchanged behavior)
- Return count: only hard-deleted (anonymous) rows counted in return value

**Test setup pattern:** Tests cannot use Better Auth's full OAuth flow for session creation. Instead, use the same approach as the existing `auth.test.ts` — use the `buildApp()` factory and simulate authenticated requests. However, for these integration tests, the simplest approach is to directly insert test users into the `users` table via `db.insert(users)`, then create a session token via `db.insert(sessions)`, and pass the session cookie in supertest requests. This mirrors what auth.test.ts does. See the existing auth test file for the pattern.

Alternatively, if direct session injection is complex, test GET /api/dashboard/secrets and DELETE by using the actual auth endpoint to create a test session (POST /api/auth/sign-in/email with test credentials), extract the session cookie from Set-Cookie, and reuse it. This is the approach used in auth.test.ts integration tests.

Use `fileParallelism: false` (already configured for server tests — no vitest.config change needed).

