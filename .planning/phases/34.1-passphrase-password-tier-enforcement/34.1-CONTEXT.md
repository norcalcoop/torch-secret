# Phase 34.1: Passphrase/Password Tier Enforcement - Context

**Gathered:** 2026-02-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Gate the passphrase/password protection options on the secret creation page by account tier. The options (No protection, Generate passphrase, Set password, Custom password) already exist — this phase restricts who can use which ones, enforces those restrictions at both the UI and API layers, and adds upgrade/signup nudges when a user hits a gate.

Anonymous: no protection options (No protection only)
Free: generated passphrase only
Pro: full access (passphrase, set password, custom password)

</domain>

<decisions>
## Implementation Decisions

### UI presentation of gated options
- All options are always visible in the combobox (never hidden)
- Pro-locked rows show a lock icon + "Pro" badge
- Clicking a Pro-locked row opens an inline popover/tooltip on that row with an upgrade CTA — the row does NOT become selected, the user's current selection stays active
- Extend the existing ARIA combobox component (same pattern as the expiration combobox established in Phase 34); locked rows are additional `role=option` items with Pro badge

### Anonymous vs. Free user distinction
- Three lock states: no lock (usable), "Free" lock (sign up to use), "Pro" lock (upgrade to use)
- Anonymous users see the passphrase row with a "Free" badge; clicking opens a signup nudge: "Create a free account to use generated passphrases" — navigates to /register
- Anonymous users see password rows with a "Pro" badge (same as Free users see)
- "No protection" is pre-selected by default for anonymous users

### Upgrade prompt design (Free → Pro)
- Clicking a Pro-locked row opens an inline popover anchored to that row
- Popover content: brief description of the Pro feature + "Upgrade →" link that navigates to /pricing
- The locked row stays unselected when the popover is open; dismissing the popover returns to whatever was previously selected

### API-layer enforcement
- Server rejects requests where protection_type doesn't match the caller's tier: 403 Forbidden with a machine-readable error code (e.g., `pro_required`, `passphrase_not_allowed`)
- Anonymous users submitting a passphrase or password field get 403 (anon = no tier = blocked) — no separate 401 guard
- Add an explicit `protection_type` enum to the POST /api/secrets schema in `shared/types/api.ts`: `'none' | 'passphrase' | 'password'` — server uses this field for tier validation rather than inferring from field presence

### Claude's Discretion
- Exact popover positioning and animation
- Specific error code strings for 403 responses
- Whether the "Free" badge and "Pro" badge are visually distinct or use the same lock icon with different text

</decisions>

<specifics>
## Specific Ideas

- The passphrase row badge for anonymous users should read "Free" (not "Sign up") — communicates what tier unlocks it, consistent with "Pro" badge on password rows
- The upgrade popover on Pro rows should link to `/pricing` (not direct checkout) — consistent with the Phase 33 Pro CTA pattern

</specifics>
