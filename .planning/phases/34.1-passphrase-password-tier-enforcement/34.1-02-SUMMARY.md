## Decisions Made
- **Popover placement outside button**: Placing `<a>` CTA links inside `<button>` elements violates axe `nested-interactive` (WCAG 4.1.2). Placing the popover div inside the tablist violates `aria-required-children` (tablist expects only tab-role children). Solution: append popovers to root container after the tablist, positioned absolutely using `getBoundingClientRect()` relative to the root.
- **tabIndex=-1 does not satisfy nested-interactive**: Axe 4.11 states that negative tabindex does not prevent assistive technologies from accessing nested interactive elements. Physical DOM placement outside the button is required.
- **Export createProtectionPanel for tests**: Since generate/custom tabs are Pro-locked in anonymous mode, tests that verify generate-tab DOM behavior (entropy line, error state) need to render a Pro-mode panel directly. Exporting the function (with JSDoc `@internal`) avoids coupling tests to auth mock infrastructure.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] ARIA nested-interactive violation: popover `<a>` nested inside tab `<button>`**
- **Found during:** Task 2 (protection panel refactor)
- **Issue:** Plan specified popover inside button element. Axe `nested-interactive` rule forbids interactive descendants inside interactive elements. `tabIndex=-1` does not satisfy the rule.
- **Fix:** Collect locked popovers in a `lockedPopovers[]` array during tab loop; append all to `root` (outside tablist) after loop completes. Set `root.style.position = 'relative'`; position popovers absolutely using `getBoundingClientRect()`.
- **Files modified:** `client/src/pages/create.ts`
