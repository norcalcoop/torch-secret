## Summary

Phase 37.2 migrates Torch Secret's 20-23 environment variables from `.env` files into Infisical Cloud (free tier) and wires every environment — local dev, GitHub Actions CI, and Render.com production — to pull secrets from Infisical at startup. No application code changes are required: the `infisical run --` CLI wrapper pattern injects secrets as environment variables before the process starts, making the integration transparent to the Node.js app.

The free tier supports up to 3 projects, 3 environments, 5 machine identities, and 10 integrations. Torch Secret needs 1 project, 3 environments (dev/staging/prod), 2 machine identities (CI + prod), and 1 integration (Render sync) — comfortably within limits. The official Infisical pricing page does not document an explicit per-project secret count limit; the "~25 secrets" limit mentioned in the CONTEXT.md should be audited against live pricing before migration.

There are three distinct integration surfaces to wire up: (1) local dev via personal `infisical login` + `.infisical.json`, (2) CI via GitHub Actions OIDC auth with `Infisical/secrets-action`, and (3) production via native Infisical → Render Secret Sync. Service tokens are deprecated — all machine identity auth must use Universal Auth (client-id + client-secret) or OIDC. The OIDC method for GitHub Actions is keyless and preferred; Universal Auth requires storing a `INFISICAL_CLIENT_SECRET` in GitHub Secrets.

**Primary recommendation:** Use OIDC auth for GitHub Actions (keyless, no secret rotation needed) and Universal Auth for the Render bootstrap credential (stored as a single Render env var). Local dev uses personal `infisical login` — no shared machine identity token needed for solo dev.

## Architecture Patterns

### Recommended Project Structure Changes
```
secureshare/
├── .infisical.json         # NEW: project-level Infisical config (workspaceId) — safe to commit
├── .env.example            # MODIFIED: key names + comments only; no values
├── .env                    # REMOVED from use: gitignored, no longer the source of truth
├── package.json            # MODIFIED: dev:server + dev:client prefixed with infisical run
└── .github/
    └── workflows/
        └── ci.yml          # MODIFIED: infisical run wraps test/build commands; OIDC auth step added
```

### Pattern 1: CLI Wrapper for npm Scripts

**What:** Prefix every command that needs env vars with `infisical run --env=<env> --`.
**When to use:** All environments — dev scripts, CI test commands, and Render start command.
**Example:**

```jsonc
// package.json scripts — AFTER migration
{
  "dev:server": "infisical run --env=dev -- tsx watch server/src/server.ts",
  "dev:client": "infisical run --env=dev -- portless torchsecret sh -c 'npx vite --port $PORT --host $HOST'"
}
```

The `--` separator is mandatory. Everything after it is the wrapped command. The `--env` flag selects which Infisical environment to pull from.

### Pattern 2: .infisical.json Project Config

**What:** A non-sensitive JSON file committed to git that binds the repo to a specific Infisical workspace (project ID).
**When to use:** Required for `infisical run` to know which project to pull from without passing `--projectId` on every command.
**Example:**

```json
// .infisical.json — safe to commit; contains no secrets
{
  "workspaceId": "<your-infisical-project-id>",
  "defaultEnvironment": "dev"
}
```

Source: Infisical CLI docs — "The `.infisical.json` file does not contain any sensitive data, so you may commit it to your git repository."

### Pattern 3: OIDC Auth for GitHub Actions (Preferred for CI)

**What:** Keyless authentication using GitHub's OIDC token — no secrets stored in GitHub.
**When to use:** GitHub Actions CI pipeline for test and lint steps.
**Example:**

```yaml
# .github/workflows/ci.yml — test job additions
permissions:
  id-token: write   # Required for OIDC token request
  contents: read

steps:
  - uses: Infisical/secrets-action@v1.0.12
    with:
      method: "oidc"
      identity-id: "${{ vars.INFISICAL_IDENTITY_ID }}"  # non-sensitive, use vars not secrets
      domain: "https://app.infisical.com"
      project-slug: "torch-secret"   # from Infisical project settings
      env-slug: "dev"                # pulls dev environment secrets

  # Test command now has all secrets as env vars:
  - name: Run tests with coverage
    run: npx vitest run --coverage
```

### Pattern 4: Universal Auth for Render (Bootstrap Credential)

**What:** Machine identity with client-id + client-secret stored as a single Render env var. Used to authenticate the Render Secret Sync, or as a fallback if native sync is insufficient.
**When to use:** Production deployment on Render.com — the one "meta-secret" that lives outside Infisical.
**Example (bootstrap token derivation):**

```bash
# One-time: generate Infisical access token from Universal Auth credentials
export INFISICAL_TOKEN=$(infisical login \
  --method=universal-auth \
  --client-id=<client-id> \
  --client-secret=<client-secret> \
  --silent --plain)
```

For the native Render sync integration, no bootstrap token is needed in the Render build command — Infisical holds a Render API key and pushes secrets directly.

### Pattern 5: Native Render Secret Sync

**What:** Infisical connects to Render via API key and automatically syncs prod secrets to Render environment variables.
**When to use:** Production environment — preferred over CLI in Render build command.
**Setup steps:**
1. In Render Account Settings > API Keys, create a Render API key
2. In Infisical: Project > Integrations > App Connections > Add Render Connection (paste API key)
3. In Infisical: Project > Integrations > Secret Syncs > Add Sync > Render
4. Source: prod environment, path `/`
5. Destination: your Render service (Service scope, not Environment Group)
6. Enable Auto-Sync

**Note on redeploy:** Render triggers a new deploy when environment variables change. Syncing from Infisical updates Render env vars, which may trigger a redeploy depending on Render service settings.

### Pattern 6: Local Dev with Personal Login

**What:** Developers authenticate with their personal Infisical account via browser-based OAuth.
**When to use:** Local development — each developer authenticates with their own account.
**Steps:**
```bash
# One-time per developer machine
infisical login              # opens browser, authenticates personal account
cd /path/to/secureshare
infisical init               # creates .infisical.json (commit this)

# Dev then works as usual:
npm run dev:server           # infisical run -- is already in the script
npm run dev:client           # same
```

### Anti-Patterns to Avoid

- **Committing `.env` with real values:** `.env` must remain gitignored; only `.env.example` (keys + comments, no values) is committed
- **Using service tokens:** Infisical deprecated service tokens in 2024 — use machine identities (Universal Auth or OIDC) instead
- **Passing `--projectId` on every infisical run call:** Use `.infisical.json` to set the workspace once and avoid repetition
- **Using `infisical run --command="..."` for the npm scripts:** Use `infisical run -- <cmd>` (double dash separator) for standard use; `--command` flag is only for chained bash commands that need `&&` or `;`
- **Fallback to `.env` file:** The locked decision is hard fail if Infisical is unreachable — do not implement a dotenv fallback
- **Removing `import 'dotenv/config'` from `server/src/config/env.ts` without testing:** The Zod env schema still validates `process.env` — infisical injects vars into the process environment before Node starts, so dotenv import becomes a no-op (harmless) rather than the source of truth; it can be removed but is not required

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Secrets injection in CI | Custom env var passing via GitHub Secrets matrix | `Infisical/secrets-action@v1.0.12` | Handles OIDC token exchange, token TTL, env export automatically |
| Render env var sync | Script that calls Render API + Infisical API | Native Infisical Render Secret Sync | UI-configured, auto-syncs on secret change, handles conflicts |
| Multi-secret CLI bulk import | Manual `infisical secrets set KEY=VAL` for each secret | `infisical secrets set --file=.env` or Infisical web dashboard UI | CLI supports `--file` flag for bulk import from `.env` format |
| OIDC token exchange | Custom JWT verification code | Infisical machine identity OIDC config | Infisical validates GitHub OIDC JWT automatically |

**Key insight:** Infisical's native integrations handle all the complexity of auth token exchange, secret versioning, and platform sync. The only glue code needed is: (1) prefix npm scripts, (2) add secrets-action step in CI, (3) configure Render sync in Infisical UI.

## Common Pitfalls

### Pitfall 1: Service Token vs Machine Identity Confusion

**What goes wrong:** Using the old service token syntax (`export INFISICAL_TOKEN=st.xxx`) which is deprecated and may stop working.
**Why it happens:** Many tutorials and Stack Overflow answers still show service token usage.
**How to avoid:** Create a machine identity with Universal Auth or OIDC. The `INFISICAL_TOKEN` env var still works but must be populated with a machine identity access token, not a service token.
**Warning signs:** Infisical CLI warnings about deprecated auth methods; auth errors after Infisical platform updates.

### Pitfall 2: Missing `--projectId` Without `.infisical.json`

**What goes wrong:** `infisical run` fails with "Missing environment or workspace id" when run in CI where `.infisical.json` doesn't exist or isn't committed.
**Why it happens:** The CLI needs to know which project to pull from; without the config file, it can't infer it.
**How to avoid:** Commit `.infisical.json` (it contains no secrets). If `.infisical.json` is committed, the CI checkout picks it up automatically. Alternatively, pass `--projectId=<id>` in every CI command.
**Warning signs:** CI failures with "Missing environment or workspace id" on the `infisical run` step.

### Pitfall 3: OIDC Subject Mismatch

**What goes wrong:** GitHub Actions OIDC auth fails with "subject mismatch" because the bound subject in Infisical doesn't match the actual GitHub Actions JWT subject.
**Why it happens:** GitHub's OIDC `sub` claim format is `repo:<owner>/<repo>:ref:refs/heads/<branch>` — the branch must match exactly.
**How to avoid:** Use glob patterns in Infisical OIDC bound subject: `repo:YourOrg/secureshare:*` to match all branches. Or set bound subject to the main branch only.
**Warning signs:** CI auth step fails with 401; Infisical audit log shows rejected OIDC token.

### Pitfall 4: Render Sync Not Triggering Redeploy

**What goes wrong:** Secrets are updated in Infisical and synced to Render, but the running service still uses old values.
**Why it happens:** Render only applies new env vars on the next deploy. Syncing updates the stored env vars but doesn't restart the service.
**How to avoid:** After syncing prod secrets, manually trigger a Render deploy or enable auto-deploy on env var changes in Render service settings.
**Warning signs:** Secret value updated in Infisical, Render sync shows green, but app behavior unchanged.

### Pitfall 5: `dotenv/config` Import Redundant but Not Harmful

**What goes wrong:** Dev assumes removing `import 'dotenv/config'` from `env.ts` is required after migration. Or conversely, assumes it breaks things.
**Why it happens:** `infisical run` injects vars into the process env before Node starts, making dotenv's file-reading redundant. But dotenv gracefully no-ops when no `.env` file exists.
**How to avoid:** Leave `import 'dotenv/config'` in place for backwards compatibility. It will attempt to load `.env` — since `.env` no longer contains real values (or doesn't exist after migration), it's a harmless no-op.
**Warning signs:** None — this is silent. The risk is if someone re-populates `.env` with real values and the app silently uses those instead of Infisical.

### Pitfall 6: Free Tier Limit on Environments (3 max)

**What goes wrong:** Creating more than 3 environments (e.g., adding `test` as a 4th) hits the free tier environment limit.
**Why it happens:** Free tier allows max 3 environments per project.
**How to avoid:** Use exactly 3 environments: dev, staging, prod. CI uses dev environment. No separate `test` environment needed since test secrets are the same as dev (placeholders).
**Warning signs:** Infisical UI shows error when attempting to create 4th environment.

### Pitfall 7: `VITE_` Prefixed Secrets Require Build-Time Injection

**What goes wrong:** `VITE_POSTHOG_KEY` and `VITE_POSTHOG_HOST` must be available at `vite build` time (Vite bakes them into the bundle), not just runtime.
**Why it happens:** Vite replaces `import.meta.env.VITE_*` references at build time using `process.env` values available to the Vite process.
**How to avoid:** The `infisical run --env=prod -- vite build` pattern covers this. In CI's `docker-build` job, ensure Infisical auth step precedes the Docker build, or pass VITE_ vars as Docker build args (current ci.yml pattern passes them as empty strings for CI builds).
**Warning signs:** Built frontend has `undefined` for PostHog key at runtime.

## Open Questions

1. **Exact free tier secret count limit**
   - What we know: Free tier shows "unlimited secrets" or no explicit cap on the pricing page; user mentioned "~25 secrets" in CONTEXT.md
   - What's unclear: The official pricing page (fetched 2026-02-27) lists environment/identity/project limits but not a secrets-per-project limit explicitly
   - Recommendation: Verify live at `infisical.com/pricing` before migration; current secret count in `.env.example` is ~23 keys — well within any reasonable free tier limit

2. **Does Render Secret Sync trigger an automatic redeploy?**
   - What we know: Syncing updates Render's stored env vars; Render triggers deploy when env vars change for services with autodeploy enabled
   - What's unclear: Whether Infisical's sync API touch triggers Render's change detection reliably
   - Recommendation: Test with a non-critical secret change first; confirm redeploy behavior before relying on it for production rotation

3. **Local dev auth: personal login vs shared machine identity**
   - What we know: Personal `infisical login` is per-developer; shared machine identity token avoids per-developer account requirement
   - What's unclear: For a solo developer project, personal login is clearly simpler; for team onboarding, machine identity token in `.env.local` would be easier
   - Recommendation: Since this is currently a solo project, use personal `infisical login`. Document both options in README so team members can choose.

4. **CI test job: inline env vars vs full Infisical injection**
   - What we know: The current `ci.yml` test job has hardcoded placeholder env vars (e.g., `LOOPS_API_KEY: loops_placeholder`). Decision is to replace these with Infisical injection.
   - What's unclear: The `docker-build` job in CI does not need app secrets (it builds with empty PostHog vars). Whether to add Infisical auth to docker-build job or leave it as-is.
   - Recommendation: Add Infisical injection only to `test` and `e2e` jobs (which need real placeholder vars). Leave `docker-build` as-is — it uses Docker `--build-arg` for VITE vars which can remain empty strings in CI.
