## Summary

Phase 37.2 migrates Torch Secret's 23 environment variables (counted from `.env.example`) from `.env` files into Infisical Cloud (free tier) and wires every environment — local dev, Docker Desktop staging, GitHub Actions CI, and Render.com production — to pull secrets from Infisical at startup. No application code changes are required: the `infisical run --` CLI wrapper pattern injects secrets as environment variables before the process starts, making the integration transparent to the Node.js app.

The free tier supports up to 3 projects, 3 environments, 5 machine identities, and 10 integrations. Torch Secret needs 1 project, 3 environments (dev/staging/prod), 2 machine identities (CI OIDC + prod Universal Auth), and 1 integration (Render sync) — comfortably within limits. The official Infisical pricing page does not document an explicit per-project secret count limit; the "~25 secrets" limit mentioned in CONTEXT.md appears to be unofficial; the current `.env.example` has approximately 23 keys, which is well within any reasonable bound.

The project's CI already has comprehensive placeholder env vars hardcoded in `ci.yml`. The locked decision replaces these with Infisical injection (OIDC auth for the `test` job). The `docker-compose.yml` file remains **unchanged** — the staging pattern wraps the `docker-compose` command at the host level, so Docker Compose forwards process-level env vars into container `environment` blocks automatically. The `dev:client` script currently uses `portless` — the Infisical prefix must wrap that entire command correctly.

**Primary recommendation:** Use OIDC auth for GitHub Actions (keyless, no secret rotation needed) and Universal Auth for the Render bootstrap credential (stored as a single Render env var). Local dev and staging use personal `infisical login` — no shared machine identity token needed for a solo developer.

The `test` job has 12 hardcoded placeholder env vars in a job-level `env:` block. The `e2e` job has 11 (missing `LOOPS_API_KEY`). The `docker-build` job has no env vars (build-time args only). The `lint` job has no env vars.

### `docker-compose.yml` app service (before migration, stays unchanged)

The app service currently has hardcoded placeholder values in the `environment:` block (e.g., `BETTER_AUTH_SECRET: 'local-development-secret-must-be-at-least-32-chars'`, `RESEND_API_KEY: 'placeholder-not-sent-in-docker-local-dev'`). When `infisical run --env=staging -- docker-compose up` is used at the host level, Docker Compose will prefer the process-level env vars (from Infisical) over the hardcoded `environment:` values in the YAML — **unless the YAML uses explicit values rather than variable references**. This is a critical distinction: Docker Compose `environment:` with literal values takes precedence over inherited process env vars; only `environment:` entries using `${VAR}` or just `VAR:` (no value) inherit from the parent process.

**Implication:** For staging injection to work, the docker-compose.yml app service `environment:` must be changed from literal values to variable references (`KEY: ${KEY}`) — OR the existing hardcoded values are acceptable for staging and only specific overrides (like `DATABASE_URL`) need to flow from Infisical. Verify this behavior in the smoke-test step of Plan 03.

## Architecture Patterns

### Recommended Project Structure Changes
```
secureshare/
├── .infisical.json         # NEW: project-level Infisical config (workspaceId) — safe to commit
├── .env.example            # MODIFIED: key names + comments only; no values
├── .env                    # REMOVED from use: gitignored, no longer the source of truth
├── package.json            # MODIFIED: dev:server + dev:client + staging:up prefixed with infisical run
└── .github/
    └── workflows/
        └── ci.yml          # MODIFIED: Infisical OIDC step added; env: blocks replaced in test + e2e jobs
```

### Pattern 1: CLI Wrapper for npm Scripts

**What:** Prefix every command that needs env vars with `infisical run --env=<env> --`.
**When to use:** All environments — dev scripts, CI test commands.
**The portless complication:** The current `dev:client` uses `portless`. The prefix wraps the entire portless invocation:

```jsonc
// package.json scripts — AFTER migration
{
  "dev:server": "infisical run --env=dev -- tsx watch server/src/server.ts",
  "dev:client": "infisical run --env=dev -- portless torchsecret sh -c 'npx vite --port $PORT --host $HOST'",
  "staging:up": "infisical run --env=staging -- docker-compose up"
}
```

The `--` separator is mandatory. Everything after it is the wrapped command. The `--env` flag selects which Infisical environment to pull from.

### Pattern 2: .infisical.json Project Config

**What:** A non-sensitive JSON file committed to git that binds the repo to a specific Infisical workspace (project ID).
**When to use:** Required for `infisical run` to know which project to pull from without passing `--projectId` on every command.
**Example:**

```json
// .infisical.json — safe to commit; contains no secrets
{
  "workspaceId": "<your-infisical-project-id>",
  "defaultEnvironment": "dev"
}
```

Source: Infisical CLI docs — "The `.infisical.json` file does not contain any sensitive data, so you may commit it to your git repository."

### Pattern 3: OIDC Auth for GitHub Actions (Preferred for CI)

**What:** Keyless authentication using GitHub's OIDC token — no secrets stored in GitHub.
**When to use:** GitHub Actions CI pipeline for `test` and `e2e` jobs.
**Important:** `DATABASE_URL` and `REDIS_URL` in CI point to GitHub Actions service containers (`localhost:5432`, `localhost:6379`) — these are CI-specific and must NOT come from Infisical. Keep them as job-level `env:` overrides alongside the Infisical step.

```yaml
# .github/workflows/ci.yml — test job additions
permissions:
  id-token: write   # Required for OIDC token request
  contents: read

steps:
  - uses: actions/checkout@v4

  - uses: Infisical/secrets-action@v1.0.12
    with:
      method: "oidc"
      identity-id: "${{ vars.INFISICAL_IDENTITY_ID }}"  # non-sensitive, use vars not secrets
      domain: "https://app.infisical.com"
      project-slug: "torch-secret"   # from Infisical project settings
      env-slug: "dev"                # pulls dev environment secrets
      export-type: "env"

  # DATABASE_URL overrides Infisical's value with CI service container URL:
  env:
    DATABASE_URL: postgresql://secureshare:secureshare@localhost:5432/secureshare
    NODE_ENV: test
```

### Pattern 4: Universal Auth for Render (Bootstrap Credential)

**What:** Machine identity with client-id + client-secret used by the native Render Secret Sync. The Infisical → Render sync integration does not require a token in Render's build command — Infisical holds a Render API key and pushes secrets directly.
**When to use:** Production deployment on Render.com — the one "meta-secret" that lives outside Infisical.

For the native Render sync integration, no bootstrap token is needed in the Render build command — Infisical holds a Render API key and pushes secrets directly.

### Pattern 5: Native Render Secret Sync

**What:** Infisical connects to Render via API key and automatically syncs prod secrets to Render environment variables.
**When to use:** Production environment — preferred over CLI in Render build command.
**Setup steps:**
1. In Render Account Settings > API Keys, create a Render API key
2. In Infisical: Project > Integrations > App Connections > Add Render Connection (paste API key)
3. In Infisical: Project > Integrations > Secret Syncs > Add Sync > Render
4. Source: prod environment, path `/`
5. Destination: your Render service (Service scope, not Environment Group)
6. Enable Auto-Sync

**Note on redeploy:** Render triggers a new deploy when environment variables change. Syncing from Infisical updates Render env vars, which may trigger a redeploy depending on Render service settings.

### Pattern 6: Local Dev with Personal Login

**What:** Developers authenticate with their personal Infisical account via browser-based OAuth.
**When to use:** Local development and staging (Docker Desktop) — each developer authenticates with their own account.

```bash
# One-time per developer machine
infisical login              # opens browser, authenticates personal account
cd /path/to/secureshare
infisical init               # creates .infisical.json (commit this)

# Dev then works as usual:
npm run dev:server           # infisical run -- is already in the script
npm run dev:client           # same
npm run staging:up           # infisical run --env=staging -- docker-compose up
```

### Pattern 7: Staging Docker Compose Env Var Forwarding

**What:** Docker Compose inherits process-level env vars from the parent shell, but only if the YAML uses variable references rather than literal values.
**The problem:** `docker-compose.yml` app service currently uses literal values in `environment:` (e.g., `BETTER_AUTH_SECRET: 'local-development-...'`). These literal values take precedence over the parent process env vars injected by Infisical.
**Resolution options:**
- Option A: Change docker-compose.yml app service `environment:` entries to `${KEY}` references — Docker Compose will then inherit from the Infisical-injected process env. **This contradicts the "docker-compose.yml stays unchanged" decision.**
- Option B: Accept that docker-compose.yml continues to use its own hardcoded staging placeholder values for most vars, and only `DATABASE_URL` (with Docker internal hostname `db:5432`) needs to be set in Infisical's staging environment — already the plan per CONTEXT.md.
- Option C: Use `docker-compose.override.yml` for staging to override specific env vars without touching the base file.

**Recommendation (Claude's Discretion):** Option B is the pragmatic approach. The staging environment's primary purpose is to test the Docker-packaged app with the correct database hostname. The `docker-compose.yml` file itself provides acceptable placeholder values for non-critical secrets. Only `DATABASE_URL` truly needs to be different, and the staging Infisical environment sets it correctly. The smoke-test (`infisical run --env=staging -- docker-compose config`) verifies that the staging env resolves.

### Anti-Patterns to Avoid

- **Committing `.env` with real values:** `.env` must remain gitignored; only `.env.example` (keys + comments, no values) is committed
- **Using service tokens:** Infisical deprecated service tokens in 2024 — use machine identities (Universal Auth or OIDC) instead
- **Passing `--projectId` on every infisical run call:** Use `.infisical.json` to set the workspace once and avoid repetition
- **Using `infisical run --command="..."` for the npm scripts:** Use `infisical run -- <cmd>` (double dash separator) for standard use; `--command` flag is only for chained bash commands that need `&&` or `;`
- **Fallback to `.env` file:** The locked decision is hard fail if Infisical is unreachable — do not implement a dotenv fallback
- **Removing `import 'dotenv/config'` from `server/src/config/env.ts`:** Infisical injects vars before Node starts, making dotenv a harmless no-op. Leave it in place.
- **Putting `DATABASE_URL` in Infisical for CI:** The CI `DATABASE_URL` points to `localhost:5432` (GitHub Actions service container) — this must remain a job-level override, not fetched from Infisical's dev environment (which has a different host).
- **Adding Infisical injection to the `lint` job:** The lint job has no env var dependencies. Leave it as-is.
- **Adding Infisical injection to the `docker-build` job:** Docker build uses `--build-arg` for VITE vars and needs no app secrets. Leave it as-is.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Secrets injection in CI | Custom env var passing via GitHub Secrets matrix | `Infisical/secrets-action@v1.0.12` | Handles OIDC token exchange, token TTL, env export automatically |
| Render env var sync | Script that calls Render API + Infisical API | Native Infisical Render Secret Sync | UI-configured, auto-syncs on secret change, handles conflicts |
| Multi-secret CLI bulk import | Manual `infisical secrets set KEY=VAL` for each secret | `infisical secrets set --file=.env` or Infisical web dashboard UI | CLI supports `--file` flag for bulk import from `.env` format |
| OIDC token exchange | Custom JWT verification code | Infisical machine identity OIDC config | Infisical validates GitHub OIDC JWT automatically |

**Key insight:** Infisical's native integrations handle all the complexity of auth token exchange, secret versioning, and platform sync. The only glue code needed is: (1) prefix npm scripts, (2) add secrets-action step in CI, (3) configure Render sync in Infisical UI.

## Common Pitfalls

### Pitfall 1: Service Token vs Machine Identity Confusion

**What goes wrong:** Using the old service token syntax (`export INFISICAL_TOKEN=st.xxx`) which is deprecated and may stop working.
**Why it happens:** Many tutorials and Stack Overflow answers still show service token usage.
**How to avoid:** Create a machine identity with Universal Auth or OIDC. The `INFISICAL_TOKEN` env var still works but must be populated with a machine identity access token, not a service token.
**Warning signs:** Infisical CLI warnings about deprecated auth methods; auth errors after Infisical platform updates.

### Pitfall 2: Missing `--projectId` Without `.infisical.json`

**What goes wrong:** `infisical run` fails with "Missing environment or workspace id" when run in CI where `.infisical.json` doesn't exist or isn't committed.
**Why it happens:** The CLI needs to know which project to pull from; without the config file, it can't infer it.
**How to avoid:** Commit `.infisical.json` (it contains no secrets). CI checkout picks it up automatically.
**Warning signs:** CI failures with "Missing environment or workspace id" on the `infisical run` step.

### Pitfall 3: OIDC Subject Mismatch

**What goes wrong:** GitHub Actions OIDC auth fails with "subject mismatch" because the bound subject in Infisical doesn't match the actual GitHub Actions JWT subject.
**Why it happens:** GitHub's OIDC `sub` claim format is `repo:<owner>/<repo>:ref:refs/heads/<branch>` — the branch must match exactly.
**How to avoid:** Use glob patterns in Infisical OIDC bound subject: `repo:norcalcoop/secureshare:*` to match all branches.
**Warning signs:** CI auth step fails with 401; Infisical audit log shows rejected OIDC token.

### Pitfall 4: Render Sync Not Triggering Redeploy

**What goes wrong:** Secrets are updated in Infisical and synced to Render, but the running service still uses old values.
**Why it happens:** Render only applies new env vars on the next deploy. Syncing updates the stored env vars but doesn't restart the service.
**How to avoid:** After syncing prod secrets, manually trigger a Render deploy or enable auto-deploy on env var changes in Render service settings.
**Warning signs:** Secret value updated in Infisical, Render sync shows green, but app behavior unchanged.

### Pitfall 5: `dotenv/config` Import — Redundant but Not Harmful

**What goes wrong:** Dev assumes removing `import 'dotenv/config'` from `env.ts` is required after migration. Or assumes it breaks things.
**Why it happens:** `infisical run` injects vars into the process env before Node starts, making dotenv's file-reading redundant. But dotenv gracefully no-ops when no `.env` file exists.
**How to avoid:** Leave `import 'dotenv/config'` in place. It attempts to load `.env` — if `.env` no longer contains real values (or doesn't exist), it's a harmless no-op.
**Warning signs:** None — this is silent. The risk is if someone re-populates `.env` with real values and the app silently uses those instead of Infisical.

### Pitfall 6: Free Tier Limit on Environments (3 max)

**What goes wrong:** Creating more than 3 environments (e.g., adding `test` as a 4th) hits the free tier environment limit.
**Why it happens:** Free tier allows max 3 environments per project.
**How to avoid:** Use exactly 3 environments: dev, staging, prod. CI uses dev environment. No separate `test` environment needed.
**Warning signs:** Infisical UI shows error when attempting to create 4th environment.

### Pitfall 7: `VITE_` Prefixed Secrets Require Build-Time Injection

**What goes wrong:** `VITE_POSTHOG_KEY` and `VITE_POSTHOG_HOST` must be available at `vite build` time (Vite bakes them into the bundle), not just runtime.
**Why it happens:** Vite replaces `import.meta.env.VITE_*` references at build time using `process.env` values available to the Vite process.
**How to avoid:** Leave the docker-build CI job as-is (passes empty strings as build args). For production Render builds, if VITE vars are needed, they must be synced to Render env vars before the build runs — the native Render sync handles this.
**Warning signs:** Built frontend has `undefined` for PostHog key at runtime.

### Pitfall 8: Docker Compose Literal Values Override Infisical Injection

**What goes wrong:** Running `infisical run --env=staging -- docker-compose up` injects env vars at the host level, but Docker Compose `environment:` entries with literal values (not `${VAR}`) take precedence inside the container.
**Why it happens:** Docker Compose distinguishes between `KEY: value` (literal — overrides parent process) and `KEY: ${KEY}` or bare `KEY:` (inherits from parent process).
**How to avoid:** Accept that the staging docker-compose.yml uses its hardcoded placeholder values for most secrets. The critical staging-specific override is `DATABASE_URL` pointing to the Docker `db` service — set this correctly in Infisical's staging environment and update the compose file's DATABASE_URL to `${DATABASE_URL}` if injection is needed. Or accept the hardcoded value is fine for local Docker testing.
**Warning signs:** App inside Docker container uses old placeholder values instead of Infisical-injected staging secrets.

### Pitfall 9: CI e2e Job Missing `LOOPS_API_KEY`

**What goes wrong:** Current `ci.yml` e2e job is missing `LOOPS_API_KEY` which is present in the test job. This is a pre-existing bug that may cause e2e test failures if any test depends on Loops integration.
**Why it happens:** Pre-existing omission when Loops was added in Phase 37.
**How to avoid:** When replacing the e2e job's env block with Infisical injection, this is automatically fixed — Infisical's dev environment includes `LOOPS_API_KEY`.
**Warning signs:** E2e tests fail with Loops-related env var validation errors.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Service tokens for machine auth | Machine identities (Universal Auth / OIDC) | Deprecation notice: April 2024 | Service tokens still work but will stop; must use machine identities |
| Static `INFISICAL_TOKEN` in CI secrets | OIDC keyless auth via `secrets-action` | Generally available 2024 | No secret rotation needed for CI auth |
| `.env` file as secrets source | `infisical run --` wrapper | Infisical CLI v0.x+ | Zero app code changes; hard fail on Infisical unreachable |
| Manual secret sync to Render | Native Infisical → Render Secret Sync | Available on free tier 2024 | UI-configured, auto-syncs, no CLI in Render build |
| Hardcoded placeholder env vars in ci.yml | Infisical OIDC injection | This phase | CI secrets managed in one place with Infisical |

**Deprecated/outdated:**
- Service tokens (`st.xxx` format): deprecated April 2024 — Infisical will enforce removal eventually. Use machine identities.
- API Keys (personal API keys for machine access): deprecated alongside service tokens. Use machine identities.

## Open Questions

1. **Exact free tier secret count limit**
   - What we know: Free tier shows no explicit secret count cap on the pricing page; user mentioned "~25 secrets" in CONTEXT.md
   - What's unclear: Whether there is an undocumented limit
   - Recommendation: Verify at `infisical.com/pricing` before migration; current count (~23 keys) is well within any reasonable free tier limit

2. **Does Render Secret Sync trigger an automatic redeploy?**
   - What we know: Syncing updates Render's stored env vars; Render triggers deploy when env vars change for services with autodeploy enabled
   - What's unclear: Whether Infisical's sync API touch triggers Render's change detection reliably
   - Recommendation: Test with a non-critical secret change first; confirm redeploy behavior before relying on it for production rotation

3. **Local dev auth: personal login vs shared machine identity**
   - What we know: Personal `infisical login` is per-developer; for a solo project it's clearly simpler
   - Recommendation: Use personal `infisical login` for dev and staging. Document in README.

4. **CI test job: which jobs get Infisical injection?**
   - Decision: Add Infisical injection to `test` and `e2e` jobs only. Leave `lint` and `docker-build` unchanged.
   - The `DATABASE_URL` and `REDIS_URL` in CI must remain as job-level `env:` overrides (CI service container addresses are not in Infisical).
   - `LOOPS_API_KEY` gap in e2e job is fixed automatically when Infisical injection replaces the env block.

5. **Docker Compose staging env var override behavior**
   - What we know: `docker-compose.yml` app service uses literal values in `environment:` block. Docker Compose literal values take precedence over parent process env vars.
   - Recommendation: Accept the staging limitation (hardcoded placeholders in docker-compose.yml remain for most vars). The staging Infisical environment primarily provides the correct `DATABASE_URL` with Docker internal hostname. The smoke-test verifies the behavior.
