**Update `.github/workflows/ci.yml`:**

The test job and e2e job currently use inline `env:` blocks to pass placeholder app secrets. Replace these with Infisical OIDC injection using `Infisical/secrets-action@v1.0.12`.

For the `test` job:
1. Add `permissions: id-token: write` to the job (required for OIDC token request; `contents: read` already implied)
2. Add the Infisical secrets-action step BEFORE the "Install dependencies" step:

```yaml
      - name: Inject secrets from Infisical
        uses: Infisical/secrets-action@v1.0.12
        with:
          method: "oidc"
          identity-id: ${{ vars.INFISICAL_IDENTITY_ID }}
          domain: "https://app.infisical.com"
          project-slug: "torch-secret"
          env-slug: "dev"
          export-type: "env"
```

3. Remove the entire `env:` block from the test job (all 12 placeholder vars — DATABASE_URL through LOOPS_API_KEY). The Infisical action exports them into the job environment.

IMPORTANT: `DATABASE_URL` and `REDIS_URL` are infrastructure secrets that point to the CI postgres/redis services. These must remain as job-level `env:` vars because they are CI-service addresses, not real secrets stored in Infisical dev environment. Keep only these two:
```yaml
    env:
      DATABASE_URL: postgresql://secureshare:secureshare@localhost:5432/secureshare
      NODE_ENV: test
```

For the `e2e` job: apply the same pattern. Add `permissions: id-token: write`. Add the Infisical injection step before "Install dependencies". Keep only infrastructure vars in the job-level env block:
```yaml
    env:
      DATABASE_URL: postgresql://secureshare:secureshare@localhost:5432/secureshare
      REDIS_URL: redis://localhost:6379
      NODE_ENV: test
```

The `lint` and `docker-build` jobs do NOT need Infisical injection — lint has no env vars and docker-build uses empty VITE_ build args. Leave them unchanged.

The OIDC bound subject uses `repo:norcalcoop/secureshare:*` — this glob covers all branches including feature branches, so CI will authenticate regardless of which branch triggers the workflow.

**Note on INFISICAL_IDENTITY_ID:** This is stored as a GitHub Actions **variable** (not secret) since it's not sensitive. The user will add this in Plan 03. The CI yaml references it as `${{ vars.INFISICAL_IDENTITY_ID }}`. The CI run will fail auth until Plan 03 sets this, which is acceptable — Plan 03 is the verification gate.

Keep all comment lines (lines starting with `#`) exactly as-is — they explain the purpose of each key. Keep optional/commented-out keys (like `# REDIS_URL=`) unchanged. Only clear values from uncommented key=value lines.
