Update the Infisical cloud project so that dev, staging, and prod hold genuinely different credentials — not copies of the same dev values. All work happens in the Infisical UI at https://app.infisical.com — no repo files change.

Purpose: Ensure Infisical is the single source of truth with environment-appropriate values, not a dev-environment clone pushed to production.
Output: All three environments hold correct, differentiated values; CLI verification confirms no test/dev credentials in prod.
</objective>

<task type="checkpoint:human-action">
  <name>Task 2: Update SHOULD-differ secrets — rotation independence and security isolation per environment</name>
  <files>none (all changes are in Infisical UI — no repo files modified)</files>
  <action>
These secrets CAN share the same value across environments without functional breakage, but SHOULD have unique values per environment for security isolation: a breach of the dev environment should not expose production credentials.

Navigate to https://app.infisical.com > project torch-secret-28-vs.

--- GENERATE UNIQUE VALUES ---

Generate a fresh BETTER_AUTH_SECRET for each environment that does not currently have a unique value.
Run this command once per environment (three times total — or just once if all three will share the current value and you want to rotate):
  node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

Generate a fresh IP_HASH_SALT for each environment:
  node -e "console.log(require('crypto').randomBytes(24).toString('hex'))"

Record each generated value before navigating away from your terminal — you will paste them into Infisical.

--- DEV ENVIRONMENT ---
Open the "dev" environment tab.

BETTER_AUTH_SECRET
  Set to a freshly generated 32-byte hex value (from command above).

IP_HASH_SALT
  Set to a freshly generated 24-byte hex value (from command above).

RESEND_API_KEY
  Recommendation: create a separate Resend API key named "torch-secret-dev" at resend.com/api-keys.
  Using a dedicated dev key means dev email sends are billed separately and cannot accidentally hit
  production sending limits or reputation metrics.
  If you prefer to share one key for now, this is acceptable — document it.

RESEND_AUDIENCE_ID
  Create a separate Resend Audience for dev at resend.com/audiences.
  Name it "Torch Secret Dev" — this prevents dev test signups from polluting the production
  marketing subscriber list.
  Paste the dev audience ID here.

LOOPS_API_KEY
  Create a separate API key in Loops.so dashboard (Settings > API Keys).
  Name it "torch-secret-dev".
  A separate key prevents dev registrations from triggering production email sequences.

GOOGLE_CLIENT_ID / GOOGLE_CLIENT_SECRET
  If Google OAuth is configured:
  Create a separate OAuth app in Google Cloud Console for dev with callback URI:
    http://localhost:3000/api/auth/callback/google
  Set GOOGLE_CLIENT_ID and GOOGLE_CLIENT_SECRET to the dev OAuth app credentials.
  The production OAuth app will have https://torchsecret.com/api/auth/callback/google as its callback.

GITHUB_CLIENT_ID / GITHUB_CLIENT_SECRET
  If GitHub OAuth is configured:
  Create a separate GitHub OAuth App for dev:
    GitHub > Settings > Developer settings > OAuth Apps > New OAuth App
    Homepage URL: http://localhost:3000
    Callback URL: http://localhost:3000/api/auth/callback/github
  Set GITHUB_CLIENT_ID and GITHUB_CLIENT_SECRET to the dev app credentials.

--- STAGING ENVIRONMENT ---
Open the "staging" environment tab.

BETTER_AUTH_SECRET
  Set to a freshly generated 32-byte hex value (different from dev).

IP_HASH_SALT
  Set to a freshly generated 24-byte hex value (different from dev).

RESEND_API_KEY / RESEND_AUDIENCE_ID / LOOPS_API_KEY
  Staging can share dev values if you do not want to create a third set of keys.
  This is acceptable for local Docker staging (not a production environment).
  Document the decision in the SUMMARY.

--- PROD ENVIRONMENT ---
Open the "prod" environment tab.

BETTER_AUTH_SECRET
  Set to a freshly generated 32-byte hex value (different from dev and staging).
  CRITICAL: If prod currently has the same value as dev, change it now. A leaked dev secret
  would otherwise allow forging production Better Auth sessions.

IP_HASH_SALT
  Set to a freshly generated 24-byte hex value (different from dev and staging).
  A unique prod salt ensures IP hashes cannot be reverse-mapped using dev environment data.

RESEND_API_KEY
  Use the production Resend API key. If not already separate from dev, create a key named
  "torch-secret-prod" at resend.com/api-keys.

RESEND_AUDIENCE_ID
  Use the production Resend Audience ID (the live subscriber list, not the dev test list).

LOOPS_API_KEY
  Use the production Loops.so API key. If not already separate from dev, create a key named
  "torch-secret-prod" in Loops.so (Settings > API Keys).

GOOGLE_CLIENT_ID / GOOGLE_CLIENT_SECRET
  Use the production OAuth app with callback URI:
    https://torchsecret.com/api/auth/callback/google

GITHUB_CLIENT_ID / GITHUB_CLIENT_SECRET
  Use the production GitHub OAuth App with callback URI:
    https://torchsecret.com/api/auth/callback/github

--- CAN BE SHARED (document as intentional) ---
These values are safe to share across environments — no action needed:
  PORT           — 3000 in all environments; not a secret
  LOG_LEVEL      — info in all environments; not a secret
  RESEND_FROM_EMAIL — same sending address across environments is acceptable
  REDIS_URL      — if using Redis, sharing a test Redis instance across dev/staging is fine

Document the intentionally-shared keys in the SUMMARY so the decision is visible.
  </action>
