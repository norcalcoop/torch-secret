# Phase 37.2: Infiscal Free Tier Integration - Context

**Gathered:** 2026-02-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Migrate all secrets from `.env` files into Infisical and wire every environment (local dev, CI, production on Render.com) to pull secrets via the Infisical CLI. This is a fresh setup — account creation, project configuration, secret import, and integration wiring all happen in this phase. Authentication, encryption, and application logic are not changed.

</domain>

<decisions>
## Implementation Decisions

### Migration scope
- Full migration — every secret in `.env.example` moves to Infisical
- Audit secrets first before importing: count against free tier limit (~25 secrets), remove any deprecated or unused keys
- `.env.example` is kept as documentation after migration — key names and comments only, no values

### Injection method
- `infisical run --` CLI wrapper pattern for all environments (no SDK, no code changes to the app)
- Hard fail if Infisical is unreachable at startup — CLI default behavior, no fallback to `.env`

### package.json scripts
- `dev:server` and `dev:client` scripts updated to prefix with `infisical run --env=dev --`
- Consistent: same wrapper pattern in dev and production

### Environment structure
- Three Infisical environments: **dev**, **staging**, **prod**
- Dev environment holds all test/local secrets including Stripe test keys and test DATABASE_URL
- CI (GitHub Actions) authenticates to Infisical and uses the dev environment — no separate GitHub Secrets for app secrets

### Bootstrap credentials (meta-secret)
- The Infisical machine identity / service token (the one secret that lives outside Infisical):
  - **Prod:** stored as a Render.com environment variable
  - **CI:** stored as a GitHub Secret (single secret, not the app secrets themselves)

### Render integration
- Native Infisical → Render sync integration configured for the prod environment
- Secrets sync automatically to Render env vars on each deploy

### GitHub Actions integration
- Infisical GitHub Actions workflow step added to CI pipeline
- Authenticates using `INFISICAL_TOKEN` from GitHub Secrets, then `infisical run` wraps the test/build command

### Developer workflow
- README updated: replaces `.env` setup section with Infisical onboarding steps
- README steps: 1) install Infisical CLI, 2) authenticate, 3) `npm run dev` (scripts already include the wrapper)

### Claude's Discretion
- Authentication method for local dev (personal `infisical login` vs. shared machine identity token in `.env.local`)
- Whether to document a secret rotation runbook — standard practice is fine
- Exact free tier feature usage (secret versioning, audit logs) — use what's available, no explicit workflow required

</decisions>

<specifics>
## Specific Ideas

- Render.com is the production host — use the native Infisical ↔ Render integration rather than CLI in the Render build command
- "Single source of truth" was the explicit goal — one place to rotate secrets across dev, CI, and prod
- The app currently has secrets for: DATABASE_URL, BETTER_AUTH_SECRET, BETTER_AUTH_URL, APP_URL, BETTER_AUTH_TRUSTED_ORIGINS, STRIPE_SECRET_KEY, STRIPE_WEBHOOK_SECRET, STRIPE_PRO_PRICE_ID, POSTHOG_API_KEY, POSTHOG_HOST, RESEND_API_KEY, LOOPS_API_KEY, REDIS_URL (optional), NODE_ENV, PORT

</specifics>
