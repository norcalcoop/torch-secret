1. **Create `CONTRIBUTING.md`** -- welcoming, practical contributor guide. Tone: warm and inclusive (matching README tone per user decision). Structure:

   **a. Welcome section:** Brief intro welcoming contributions. "Whether you're fixing a typo, reporting a bug, or building a new feature -- thank you!"

   **b. Getting Started / Prerequisites:**
   - Node.js 24+ and npm
   - PostgreSQL 17+ (local or Docker)
   - Optional: Redis for distributed rate limiting
   - Git

   **c. Development Setup:**
   ```bash
   git clone https://github.com/norcalcoop/secureshare.git
   cd secureshare
   npm install
   cp .env.example .env  # Configure DATABASE_URL
   npm run db:migrate
   ```
   Then start dev servers:
   ```bash
   npm run dev:server    # Backend on :3000
   npm run dev:client    # Frontend on :5173 (proxies /api to :3000)
   ```
   OR use Docker:
   ```bash
   docker compose up
   ```

   **d. Code Style:**
   - ESLint 10 and Prettier 3.8 are enforced automatically
   - Pre-commit hooks (Husky + lint-staged) run lint and format on staged files
   - You can run manually: `npm run lint` and `npm run format:check`
   - TypeScript strict mode: `npx tsc --noEmit` must pass

   **e. Testing:**
   - Unit/integration tests: `npm run test:run` (Vitest, requires PostgreSQL)
   - E2E tests: `npm run test:e2e` (Playwright, requires PostgreSQL + Redis)
   - Run a specific test: `npx vitest run path/to/test.ts`
   - Client tests run in happy-dom, server tests run sequentially against real PostgreSQL

   **f. Project Structure:** Brief directory guide:
   - `client/` -- Vanilla TypeScript frontend (Vite + Tailwind)
   - `server/` -- Express 5 API server (Drizzle ORM + PostgreSQL)
   - `shared/` -- Zod schemas and TypeScript types shared between client and server
   - `e2e/` -- Playwright end-to-end tests
   - `drizzle/` -- Database migration files

   **g. Making a Pull Request:**
   1. Fork the repository
   2. Create a feature branch (`git checkout -b my-feature`)
   3. Make your changes
   4. Run the full quality check: `npm run lint && npm run format:check && npm run test:run && npx tsc --noEmit`
   5. Commit with a descriptive message
   6. Push and open a PR against `main`
   7. CI will run lint, tests, and E2E automatically

   **h. Reporting Bugs:** Link to the bug report issue template.

   **i. Security:** For security vulnerabilities, see [SECURITY.md](SECURITY.md). Do not file public issues for security concerns.

2. **Create `SECURITY.md`** -- Security policy and responsible disclosure:
   - Title: "Security Policy"
   - **Supported Versions:** Table showing v3.x as supported, v2.x and v1.x as end-of-life
   - **Reporting a Vulnerability:**
     - Preferred: Use GitHub's private vulnerability reporting at `https://github.com/norcalcoop/secureshare/security/advisories/new`
     - Describe the vulnerability, steps to reproduce, and potential impact
     - Expected response time: within 48 hours
     - Do NOT file public issues for security vulnerabilities
   - **Security Model:** Brief note: SecureShare uses a zero-knowledge architecture. The server never has access to plaintext or encryption keys. For architecture details, see the README.
