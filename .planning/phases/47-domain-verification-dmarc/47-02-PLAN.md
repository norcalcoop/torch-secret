<task type="checkpoint:human-action">
  <name>Task 1: Verify torchsecret.com in Resend and Loops.so dashboards</name>
  <action>
    Wait at least 5 minutes after Plan 01 completing before triggering verification. If Plan 01 dig checks all passed, propagation is sufficient.

    **Step A: Verify Resend domain**
    1. Open Resend dashboard → Domains → torchsecret.com
    2. Click "Verify Domain" (or the equivalent verify button on the domain detail page)
    3. Wait for the per-record status to update — the page shows individual status for DKIM, SPF, and MX
    4. All three must show green / "Verified"

    If verification fails on first attempt:
    - Wait 15 minutes and retry — SES polling takes time even after Cloudflare propagates
    - If still failing after 30 minutes: run dig checks from Plan 01 Task 2 verify section to confirm records are still resolving correctly
    - Common cause: DKIM TXT value was copied with trailing whitespace or line breaks — verify the raw TXT record in Cloudflare matches exactly

    **Step B: Verify Loops.so domain**
    1. Open Loops.so dashboard → Settings → Domain
    2. Click "Verify Records" (button wording may differ slightly — look for a verify or refresh action)
    3. Wait for per-record status — all DKIM, SPF, and MX records should show green

    If verification fails on first attempt:
    - Wait 15 minutes and retry
    - Most common cause for Loops: a DKIM CNAME is proxied (orange cloud). Run: `dig [loops-selector]._domainkey.torchsecret.com CNAME +short` — if it returns a Cloudflare IP instead of amazonaws.com, return to Cloudflare DNS, find the CNAME, toggle it to DNS Only (grey cloud), save, wait 5 min, retry verify

    **Step C: Confirm hello@torchsecret.com as Loops sender**
    1. In Loops.so dashboard → Settings → Domain (or the sender address/From address section)
    2. Confirm the From address for campaigns is set to `hello@torchsecret.com`
    3. If it shows a different address (e.g., a @loops.so default), update it to hello@torchsecret.com
    4. Save the setting
  </action>
  <verify>
    Resend: Dashboard shows torchsecret.com with status "Verified" and DKIM / SPF / MX all green.
    Loops: Dashboard shows torchsecret.com domain as verified (all records green).
    Loops: From address in Settings confirms hello@torchsecret.com.
  </verify>
  <done>
    Resend shows torchsecret.com as Verified. Loops.so shows torchsecret.com domain as verified. Loops From address is set to hello@torchsecret.com. RSND-01, LOOP-01, LOOP-02 satisfied.
  </done>

<task type="checkpoint:human-action">
  <name>Task 2: Send Resend API test email and confirm inbox delivery</name>
  <action>
    With Resend verification complete from Task 1, send a test email from noreply@torchsecret.com to confirm end-to-end delivery.

    **Step A: Get RESEND_API_KEY from Infisical**
    Run the following to retrieve the key (this is the existing production key — do NOT create a new one):
    ```bash
    infisical secrets get RESEND_API_KEY --env=production --plain
    ```
    Copy the key value.

    **Step B: Send test email via Resend API**
    ```bash
    curl -X POST 'https://api.resend.com/emails' \
      -H 'Authorization: Bearer YOUR_RESEND_API_KEY' \
      -H 'Content-Type: application/json' \
      -d '{
        "from": "Torch Secret <noreply@torchsecret.com>",
        "to": ["torch-secret@gmail.com"],
        "subject": "Resend domain verification test — noreply@torchsecret.com",
        "text": "This email confirms noreply@torchsecret.com is an authenticated sending address via Resend. Domain verification is complete."
      }'
    ```

    Replace `YOUR_RESEND_API_KEY` with the actual key.

    Expected success response: `{"id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"}`

    If the response is `{"statusCode": 403, "message": "The 'noreply@torchsecret.com' from address is not allowed..."}`: Resend verification from Task 1 is not yet complete — return to Task 1 and confirm the Verified status before retrying.

    **Step C: Check Gmail inbox**
    1. Open torch-secret@gmail.com inbox
    2. Confirm the email from `noreply@torchsecret.com` arrived in the inbox (not spam)
    3. Open the email and verify the From address shows `noreply@torchsecret.com` (not an @resend.dev or @amazonses.com address)

    If the email lands in spam:
    - Mark it "Not spam"
    - Check the email headers (Show original in Gmail) — confirm DKIM pass and SPF pass
    - This is a reputation issue with a freshly verified domain; DKIM/SPF passing is what matters for this phase

    **Step D: Optional — Resend dashboard "Send test" button**
    In Resend dashboard → Domains → torchsecret.com, use the built-in "Send test" button as an additional smoke test. This confirms the dashboard flow works independently of the curl API call.
  </action>
  <verify>
    curl returns `{"id": "..."}` (not a 403 error).
    torch-secret@gmail.com receives the email from noreply@torchsecret.com.
    Email arrives in inbox (or is manually moved from spam if it lands there — DKIM/SPF pass in headers is the authoritative check).
  </verify>
  <done>
    Resend API test email delivered from noreply@torchsecret.com to torch-secret@gmail.com. Email headers show DKIM=pass and SPF=pass. Phase 47 all four success criteria satisfied:
