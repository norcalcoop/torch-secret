<objective>
Close the BILL-05 race window: the verify-checkout endpoint currently confirms Stripe payment but does NOT persist Pro status to the DB, relying entirely on webhook timing. This plan makes activatePro() idempotent (it already is structurally — DB UPDATE + Loops updateContact are both idempotent) and calls it from the verify-checkout handler so that Pro activation happens immediately when the user returns from Stripe Checkout.

Purpose: Eliminate the ~1-2s window where a user returns from Checkout, the dashboard calls verify-checkout, gets `{ status: 'active', tier: 'pro' }`, but the DB still has `subscriptionTier: 'free'` because the webhook hasn't arrived yet. After this plan, verify-checkout writes Pro status to DB directly.

Output: Updated billing.ts with activatePro() call in verify-checkout, new billing.test.ts route tests, extended billing.service.test.ts with idempotency test.

```typescript
billingRouter.get('/verify-checkout', requireAuth, async (req, res) => {
  const user = res.locals.user as AuthUser;
  const sessionId = req.query['session_id'] as string | undefined;

  if (!sessionId?.startsWith('cs_')) {
    res.status(400).json({ error: 'invalid_session_id' });
    return;
  }

  const [dbUser] = await db.select().from(users).where(eq(users.id, user.id));
  const session = await stripe.checkout.sessions.retrieve(sessionId);

  if (session.status !== 'complete') {
    res.status(402).json({ error: 'payment_incomplete' });
    return;
  }

  // Verify the session belongs to this user (CSRF/spoofing protection).
  if (dbUser?.stripeCustomerId && session.customer !== dbUser.stripeCustomerId) {
    res.status(403).json({ error: 'session_mismatch' });
    return;
  }

  res.json({ status: 'active', tier: 'pro' });  // ← GAP: does not call activatePro()
});
```
