## Tech Debt

### Phase 22: auth-page-noindex (low severity)

**Item:** Auth routes (`/login`, `/register`, `/forgot-password`, `/reset-password`, `/dashboard`) receive noindex only via client-side `updatePageMeta({ noindex: true })` calls in router.ts. The server-side `X-Robots-Tag: noindex, nofollow` header in app.ts only applies to paths starting with `/secret/`.

**Impact:** Crawlers that skip JavaScript will not see noindex on auth pages and may index them. No auth requirement (AUTH-01 through AUTH-08) is violated. No security implication (pages are publicly accessible by design). Minor SEO concern only.

**Recommendation:** In a future hardening pass, extend the server-side X-Robots-Tag guard in app.ts to also cover `/login`, `/register`, `/forgot-password`, `/reset-password`, and `/dashboard`.

### Phase 22: human-verification-pending (informational)

**Item:** 7 live-infrastructure behavioral flows are pending human verification with real credentials:
1. Registration + email delivery round-trip (AUTH-01, AUTH-02)
2. Email verification gate for unverified login (AUTH-02)
3. Session persistence across browser refresh (AUTH-05)
4. Password reset full email round-trip (AUTH-04)
5. Logout via dashboard browser button (AUTH-08)
6. Google OAuth round-trip (AUTH-06)
7. GitHub OAuth round-trip (AUTH-07)

**Impact:** Implementation is complete and all 19 automated checks pass. These flows require a live Resend API key, real browser with cookie handling, and OAuth credentials. Not a code defect â€” a testing infrastructure constraint.

**Recommendation:** Perform manual QA against the production environment when RESEND_API_KEY, GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET, GITHUB_CLIENT_ID, GITHUB_CLIENT_SECRET are provisioned. All 7 flows are already documented in the Phase 22 VERIFICATION.md human-verification section.
